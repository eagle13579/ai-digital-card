# 链客宝微信小程序 —「通讯录导入」功能产品需求文档

> 版本：v1.0  
> 状态：草案  
> 关联模块：AI数智名片、联系人管理、数据隐私中心

---

## 1. 背景与目标

### 1.1 用户痛点

| 痛点 | 描述 |
|------|------|
| **名片维护成本高** | 企业家用户手机通讯录中累积了 500–2000+ 联系人，逐一手动录入 AI 数智名片耗时费力，很多用户因此放弃录入 |
| **联系人碎片化** | 同一个客户的联系方式分散在微信聊天记录、手机通讯录、纸质名片等多个载体，缺乏统一管理入口 |
| **关系价值沉睡** | 大量联系人仅存手机号/微信号，用户无法快速检索对方的公司、职位、行业标签，商务线索被动流失 |
| **AI 名片缺乏初始数据** | 新用户注册后 AI 数智名片为空，冷启动阶段无法提供智能推荐、人脉分析等核心价值，导致留存率低 |

### 1.2 核心目标

**一句话目标**：通过一键导入手机通讯录，让用户在 30 秒内完成联系人批量录入，消除手动维护负担，同时为 AI 名片积累结构化人脉数据。

**三个关键指标（KPI）**：

| 指标 | 目标值 | 测量方式 |
|------|--------|----------|
| 通讯录导入完成率 | ≥ 65%（发起导入的用户中完成全流程的比例） | 埋点事件：发起导入 → 授权同意 → 导入成功 |
| 导入后次日留存 | 导入用户次日留存率较未导入用户提升 ≥ 20% | 用户行为分析 |
| 平均导入联系人数量 | 单次导入 ≥ 200 人 | 导入完成事件上报数据量 |

### 1.3 成功标准

- **MVP 成功标准**：用户在 3 步内完成导入（授权 → 选择联系人 → 确认导入），导入成功率达到 90% 以上（即发起后无中途退出）
- **质量成功标准**：导入的联系人中，手机号或微信号匹配准确率 ≥ 95%（允许 5% 以内的号码格式异常标记为待修正）
- **合规成功标准**：通过微信小程序隐私审核，无隐私投诉工单
- **业务成功标准**：上线 30 天内，至少 40% 的日活用户完成通讯录导入

## 2. 用户场景与用户故事

### 2.1 典型用户场景

#### 场景 A：新用户冷启动

> 张总是一家制造企业的 CEO，刚注册链客宝，系统提示「上传名片或导入通讯录」来完善人脉。他点击「导入手机通讯录」，弹出系统权限请求，同意后看到 350 个联系人被识别，其中 80 人已注册链客宝并自动关联名片，其余 270 人作为「待邀请」联系人存入名片夹。全程不到 20 秒。

**关键触发点**：注册成功后的引导页、首页空态提示

#### 场景 B：老用户批量更新

> 李经理使用链客宝 3 个月，手机通讯录新增了 50 个商务联系人。她打开「我的联系人」页面，看到「通讯录有更新，导入新增联系人」的提示条，点击后系统只增量导入新人，已有名片的关系不受影响。

**关键触发点**：联系人页面顶部提示条、定期（每 7 天）检测变化

#### 场景 C：定向邀请与匹配

> 王总在链客宝搜索客户「陈总」，系统提示「未找到，你可能认识以下联系人」，列出了通讯录中名字相似的未注册联系人。王总点击「邀请加入」，通过微信模板消息发送邀请链接，陈总注册后两人自动建立企业关系。

**关键触发点**：搜索无结果时的联系人推荐、通讯录「可能认识的人」模块

#### MVP 验收标准

```
F1-导入入口
  Given 用户未导入过通讯录
  When  用户进入「我的联系人」页面
  Then  页面顶部展示「导入通讯录」入口按钮，点击后跳转授权流程
  And   注册后 24 小时内，首页至少展示 1 次导入引导弹窗

F2-用户授权
  Given 用户点击「导入通讯录」
  When  系统检查 wx.chooseContact 可用性
  Then  可用时调用 wx.chooseContact → 用户选择联系人
  And   不可用时（低版本微信）展示文件导入备选入口

F3-数据预览
  Given 用户完成联系人选择
  When  系统解析并展示联系人列表
  Then  每项展示：头像（如有）、姓名、手机号（后 4 位显示，其余*隐藏）、微信号（如有）
  And   每项左侧有 checkbox，默认全选
  And   底部显示「已选 N 人 / 共 M 人」
  And   用户可取消勾选任意项

F4-导入执行
  Given 用户点击「确认导入」
  When  系统批量写入 digital_brochure.db
  Then  以手机号为唯一约束，已存在记录跳过
  And   已注册链客宝的联系人自动建立名片关联
  And   导入成功后返回结果页
  And   前端 loading 时长 ≤ 3 秒（1000 条数据内）

F5-结果展示
  Given 导入完成
  When  进入结果页面
  Then  展示三组数据：已导入 X 人 / 已注册 Y 人 / 待邀请 Z 人
  And   已注册联系人可点击「加好友」
  And   待邀请人可点击「邀请加入」

F6-数据管理
  Given 用户有已导入的联系人
  When  在联系人管理页操作删除
  Then  支持全选、反选、单条删除
  And   删除后对方不再出现在用户的名片夹中
  And   数据库标记软删除（is_deleted=1），保留 30 天可恢复
```

## 4. 隐私与合规

### 4.1 微信小程序隐私政策要求

| 要求 | 实现方式 |
|------|----------|
| **显式授权** | 必须在 `wx.chooseContact()` 调用前弹出独立的《通讯录数据使用授权协议》，用户点击「同意」后方可调用 API |
| **最小 API 调用** | 优先使用 `wx.chooseContact()`（用户主动选择），不使用 `wx.getPhoneNumber()` 自动获取 |
| **用途声明** | 在《用户隐私政策》中单独章节声明通讯录数据的收集目的、范围、存储期限、共享情况 |
| **拒绝不影响基本功能** | 用户拒绝通讯录授权后，链客宝的名片创建、名片交换等核心功能仍可正常使用，仅「批量导入」功能不可用 |
| **审核材料准备** | 提前准备：隐私政策截图、授权弹窗截图、数据流转说明、服务器端数据加密方案 |

#### 隐私授权弹窗文案（参考）

```
标题：导入通讯录联系人
正文：链客宝需要读取你选择的联系人信息（姓名、手机号、微信号），用于：
  · 将联系人存入你的AI数智名片夹
  · 匹配已注册链客宝的用户并建立连接
  · 基于联系人推荐你可能认识的人
我们不会存储联系人的微信号，不会将数据分享给第三方，不会在未授权的情况下主动向联系人发送信息。
你可以随时在「设置→隐私中心」删除已导入的联系人。
  [拒绝]  [同意并导入]
```

### 4.3 数据最小化原则

| 数据字段 | 是否存储 | 用途 | 说明 |
|---------|---------|------|------|
| 姓名 | ✅ 存储 | 名片显示、搜索索引 | 必存字段 |
| 手机号 | ✅ 存储 | 唯一标识、已注册匹配、邀请链接 | 导入后脱敏展示（显示尾号 4 位） |
| 微信号 | ❌ 不存储 | — | wx.chooseContact 返回即丢弃，仅用于当前匹配校验 |
| 头像 URL | ✅ 可选 | 联系人卡片展示 | 若 API 返回则存储 |
| 备注名 | ✅ 可选 | 用户自定义别名 | 用户可在导入后编辑 |
| 联系人分组 | ❌ 不存储 | — | 微信分组信息不在本功能使用范围内 |
| 通讯录完整列表 | ❌ 不上传 | — | 仅处理用户主动选择的联系人，不上传完整通讯录 |

> **关键约束**：本功能不使用 `wx.getPhoneNumber()`（自动获取微信绑定手机号），只使用 `wx.chooseContact()`（用户主动选择）——这是微信小程序隐私合规的最低要求路径。

### 4.4 用户删除数据的能力

| 删除方式 | 说明 | 实现 |
|---------|------|------|
| **单条删除** | 在联系人详情页点击「删除」 | 软删除（is_deleted=1），30 天内可在「回收站」恢复 |
| **批量删除** | 在联系人管理页全选/选择多条后删除 | 同上 |
| **清空所有导入联系人** | 在「设置→隐私中心」点击「清空导入的通讯录数据」 | 硬删除所有 is_deleted=0 记录，同时取消已建立的名片关联 |
| **注销账户时自动清除** | 用户注销链客宝账户时连带清除 | 触发 cascade delete，所有导入数据硬删除 |
| **回收站机制** | 软删除的数据 30 天后自动物理清理 | 定时任务：`DELETE FROM contacts WHERE is_deleted=1 AND deleted_at < NOW() - INTERVAL 30 DAY` |

**用户可见的删除入口**：
- 联系人列表 → 长按/右滑 → 删除
- 联系人列表 → 右上角「管理」→ 批量删除
- 我的 → 设置 → 隐私中心 → 通讯录数据管理 → 清空已导入数据

### 4.5 GDPR / 个保法合规要点

| 合规原则 | 实现要求 | 具体措施 |
|---------|---------|---------|
| **知情权 (§13/Art.13)** | 数据收集前告知用户处理目的、范围、期限 | 授权弹窗 + 隐私政策全文中单独章节 |
| **同意权 (§14/Art.7)** | 获取明示、自由作出的同意 | 弹窗勾选「同意」不能默认预勾选，不能将同意作为使用小程序的唯一前提 |
| **删除权 (被遗忘权) (§47/Art.17)** | 用户有权要求删除其个人数据 | 提供清晰的删除入口 + 账户注销自动清理 |
| **数据最小化 (§6/Art.5(1c))** | 只收集实现目的所必需的数据 | 不收集微信号，不上传完整通讯录，仅处理用户主动选择的数据 |
| **目的限制 (§6/Art.5(1b))** | 数据仅用于声明目的 | 导入数据仅用于名片夹匹配和关系推荐，不得用于未经声明的商业化用途 |
| **数据安全 (§51/Art.32)** | 采取技术措施保护个人数据 | 传输加密（TLS 1.3）、存储加密（AES-256）、手机号脱敏展示 |
| **跨境传输 (§38/Art.44–49)** | 数据不出境或取得合规授权 | 初期部署在国内服务器（阿里云/腾讯云），如后续使用海外服务需单独评估 |
| **响应时限 (§49/Art.12(3))** | 对用户的数据主体请求在 15 个工作日内（个保法）/ 1 个月内（GDPR）响应 | 建立隐私工单处理流程 |

#### 隐私政策补充章节（建议结构）

```
## 4. 通讯录数据的收集与使用

### 4.1 我们收集哪些数据
- 你在使用「通讯录导入」功能时主动选择的联系人信息（姓名、手机号）
- 导入时间戳
- 导入结果状态

### 4.2 我们如何使用这些数据
- 存入你的 AI 数智名片夹，供你查看和管理
- 与链客宝已注册用户进行名片匹配
- 基于你的联系人推荐你可能认识的人

### 4.3 我们不做什么
- 不上传你的完整通讯录
- 不存储你联系人的微信号
- 不向第三方出售或分享这些数据
- 不在未经你授权的情况下向你的联系人发送消息

### 4.4 数据存储与删除
- 数据存储于国内服务器
- 你可以随时删除单个或全部导入的联系人
- 账户注销时所有数据一并清除
- 软删除的数据 30 天后自动物理清理
```

| created_at | DATETIME | 创建时间 | DEFAULT CURRENT_TIMESTAMP |
| deleted_at | DATETIME | 删除时间（软删除时写入） | NULLABLE |

### B. 未来 PostgreSQL 迁移注意事项

- 使用 `UNIQUE(user_id, phone)` 约束保证去重，迁移后确认 PostgreSQL 的 UNIQUE 组合索引行为一致
- `is_registered` 字段可改为从 `registered_user_id IS NOT NULL` 推导，消除冗余
- 手机号字段建议使用 `VARCHAR` 而非 `BIGINT`，避免前导零丢失问题
- 数据量达到 10 万级时，为 `(user_id, is_deleted)` 建立复合索引以优化查询性能
- 软删除的物理清理使用 PostgreSQL 的 `pg_cron` 定时任务实现
