# 架构决策记录 (Architecture Decision Records — ADR)

## 什么是 ADR？

ADR（Architecture Decision Record）是记录重大架构决策的轻量级文档。每个 ADR 记录一次关键决策的背景、方案选择、理由和影响，帮助团队理解"为什么这么做"。

## 编号规则

- 格式：`ADR-XXX`，例如 `ADR-001`、`ADR-042`
- 编号按创建顺序递增，**永不重用**
- 即使某条 ADR 被废弃，其编号也永久保留

## 状态流

```
                  ┌──────────────┐
                  │  Proposed    │  (草案，正在讨论中)
                  └──────┬───────┘
                         │ 批准
                         ▼
                  ┌──────────────┐
                  │  Accepted    │  (已采纳，当前有效)
                  └──────┬───────┘
                        ╱ ╲
                       ╱   ╲
              被新决策替代   不再适用
                     ▼           ▼
             ┌──────────────┐  ┌──────────────┐
             │  Superseded  │  │  Deprecated  │
             │ (被 XXX 替代)│  │ (已废弃)      │
             └──────────────┘  └──────────────┘
```

### 状态说明

| 状态 | 含义 |
|------|------|
| **Proposed** | 提案阶段，团队正在评审和讨论 |
| **Accepted** | 已通过评审，当前有效的架构决策 |
| **Deprecated** | 不再适用，但未指定替代方案 |
| **Superseded** | 被更新的 ADR 明确替代（需注明替代者编号） |

## ADR 文件规范

- 文件命名：`ADR-XXX-简短描述.md`（英文或拼音/中文均可）
- 存放位置：`backend/docs/adr/`
- 每条 ADR 应包含以下章节：

```
标题: 简明扼要的决策主题
状态: Proposed | Accepted | Deprecated | Superseded
日期: YYYY-MM-DD
决策者: 参与决策者名单

## 上下文
描述需要做决策的背景、问题、约束条件。

## 决策
明确说明选择了什么方案。

## 理由
解释为什么选择了这个方案而非其他方案。

## 后果
接受此决策带来的正面和负面影响。

## 替代方案
列出被否决的方案及否决理由。
```

## 如何新增 ADR

1. 从最新 ADR 编号 +1 获得新编号
2. 复制模板并填写
3. 提交 PR 供团队评审
4. 评审通过后将状态改为 `Accepted`
5. 更新 `backend/docs/INDEX.md`
