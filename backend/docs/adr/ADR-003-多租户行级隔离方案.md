# ADR-003: 多租户行级隔离方案

**状态**: Accepted
**日期**: 2026-06-26
**决策者**: 架构组、后端组

## 上下文

链客宝采用多租户架构（SaaS 模式），一个 PostgreSQL 实例服务于多个企业客户。每个租户的数据必须严格隔离，防止租户 A 看到租户 B 的数据。

隔离方案有三个方向：
- **方案 A**：每个租户独立数据库（Database per Tenant）
- **方案 B**：每个租户独立 Schema（Schema per Tenant）
- **方案 C**：同一数据库 + 同一 Schema，通过 `tenant_id` 列做行级隔离

## 决策

采用 **方案 C：行级隔离 + 自动注入 `tenant_id` 中间件**。

核心设计：

1. **统一租户标识字段**：所有业务表必须包含 `tenant_id: UUID` 列，且为非空、有索引
2. **`tenant_middleware.py`**：FastAPI 中间件从 JWT Token 中提取 `tenant_id`，注入到 `request.state.tenant_id`
3. **SQLAlchemy 自动过滤**：通过 `@tenant_aware` 装饰器或自定义查询基类，自动在 SQL 查询中添加 `WHERE tenant_id = :tid`
4. **超级管理员**：携带 `scope:admin` 的 Token 可跨越租户查询
5. **数据导入/导出**：支持按租户导出全量数据

关键代码模式：

```python
# tenant_middleware.py
@app.middleware("http")
async def add_tenant_id(request: Request, call_next):
    token = request.headers.get("Authorization", "").removeprefix("Bearer ")
    payload = decode_jwt(token)
    tenant_id = payload.get("tenant_id")
    if tenant_id:
        request.state.tenant_id = tenant_id
    return await call_next(request)
```

```python
# models/base.py
class TenantBase(Base):
    __abstract__ = True
    tenant_id: Mapped[UUID] = mapped_column(ForeignKey("tenants.id"), nullable=False, index=True)
```

## 理由

1. **运维简单** — 一个数据库实例，无需动态创建数据库或 Schema，备份和迁移方便
2. **跨租户查询可行** — B2B 匹配平台的核心场景是两个不同租户的企业互相匹配，行级隔离+显式授权可实现跨租户访问
3. **资源效率高** — 1000 个租户共用一个连接池，而不是每个租户独立连接池，数据库连接数可控
4. **SQLAlchemy 天然支持** — 通过 `@validates('tenant_id')` 或自定义 `Query` 类的 `filter_by` 可以零侵入地实现自动过滤

## 后果

**正面**:
- 单个数据库实例即可支持数百个租户，基础设施成本低
- 跨租户匹配功能（核心业务）实现简单
- 备份/恢复/迁移操作只需处理一个数据库

**负面**:
- 必须严格保证每个查询都带上 `tenant_id` 过滤，遗漏可能导致数据泄露
- 数据库表行数随租户数线性增长，大租户可能影响小租户的查询性能（需要 pg_stat_statements 监控）
- 无法像独立数据库那样做租户级别的备份恢复

## 替代方案

| 方案 | 否决理由 |
|------|---------|
| Database per Tenant | 1000 个数据库连接池耗尽、迁移脚本需执行 1000 次、跨租户查询困难 |
| Schema per Tenant | PostgreSQL Schema 数量过千时性能下降，ORM 迁移工具对多 Schema 支持不完善 |
| 应用层加密存储 | 查询无法利用索引和数据库排序，搜索功能基本不可用 |
