# ADR-002: 7 层纵深防御安全架构

**状态**: Accepted
**日期**: 2026-06-26
**决策者**: 安全组、架构组

## 上下文

链客宝作为 B2B 平台，存储企业敏感信息（工商资料、联系方式、匹配意向等），面临以下安全威胁：

- SQL 注入 / NoSQL 注入
- XSS / CSRF
- 未授权访问 / 越权操作
- API 滥用 / DDoS
- 数据泄露（内部或外部）
- JWT Token 窃取 / Session 劫持

需要设计一套中间件安全架构，在不影响性能的前提下提供纵深防御。

## 决策

采用 **7 层中间件链** 的安全架构，按请求处理顺序依次为：

```
请求进入
   │
   ▼
① Rate Limiter (限流)        — 基于 Redis 滑动窗口，按 IP + UserID 限流
   │
   ▼
② CORS / CSP Headers        — 严格 CORS 策略 + Content-Security-Policy
   │
   ▼
③ Request Validator         — Pydantic 校验 + SQL 关键字过滤 + 输入清洗
   │
   ▼
④ JWT Auth Middleware       — JWT 双 Token 机制 (Access + Refresh)，自动续签
   │
   ▼
⑤ Role & Permission Check   — RBAC 角色权限验证（装饰器 + 依赖注入）
   │
   ▼
⑥ Tenant Isolation Check   — 多租户行级隔离（见 ADR-003）
   │
   ▼
⑦ Audit Logger             — 敏感操作审计日志（写操作记录到 audit_log 表）
   │
   ▼
   路由 → Controller
```

每层部署为独立的 FastAPI `Middleware` 或 `Dependency`，可单独启用/禁用，支持单元测试。

## 理由

1. **纵深防御** — 单层防御可能被绕过，7 层确保即使某一层失效，后续层仍可拦截攻击。
2. **依赖注入天然适配** — FastAPI 的 `Depends()` 机制可以精确控制哪些路由需要哪些安全层，公共路由（如登录）跳过 JWT 验证但仍受 Rate Limiter 保护。
3. **可观测性** — 每层都有独立的日志和指标输出，方便安全审计和故障定位。
4. **渐进式性能影响** — 轻量层（CORS、Validator）在前，重量层（JWT 解码、RBAC 查库）在后，尽早拒绝无效请求。

## 后果

**正面**:
- 安全审计链路清晰，每条安全策略可单独测试
- 新增安全层只需在中间件链中插入，不影响业务代码
- 满足等保二级 / SOC2 的部分合规要求

**负面**:
- 每个请求经过 7 层检查，增加约 3-8ms 延迟（实测）
- RBAC 角色查库可能成为瓶颈（需用 Redis 缓存角色数据）
- 中间件顺序依赖性强，调错顺序可能导致安全漏洞

## 替代方案

| 方案 | 否决理由 |
|------|---------|
| Nginx 层完成所有安全 | Nginx 无法做业务级 RBAC 和租户隔离，且 Lua 脚本维护成本高 |
| 单一全局中间件 | 耦合严重，新增安全策略困难，不符合单一职责原则 |
| 第三方安全网关 (WAF) | 成本高，且无法感知业务上下文（如租户 ID），仍需应用层补充 |
