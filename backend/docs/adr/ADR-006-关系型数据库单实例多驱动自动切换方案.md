# ADR-006: 关系型数据库单实例多驱动自动切换方案

**状态**: Accepted
**日期**: 2026-06-26
**决策者**: 架构组
**关联 ADR**: ADR-001（FastAPI + SQLAlchemy + 三驱动数据库架构）

## 上下文

在 ADR-001 中确立了链客宝整体数据存储架构。对于核心业务关系型数据，我们需要在不同环境下使用不同的数据库驱动，以平衡开发便利性和生产性能：

1. **本地开发环境 (Dev)**：使用 SQLite。优点是零配置、单文件、即开即用，方便开发者在本地快速启动和调试，无需安装数据库服务。
2. **预发布环境 (Staging)**：使用 MySQL。模拟生产环境的主流关系型数据库，验证 SQL 兼容性和事务行为。
3. **生产环境 (Production)**：使用 PostgreSQL。提供更强的事务一致性、更丰富的数据类型（如 JSONB, UUID）和更优的复杂查询性能。

**核心挑战**：如何在不修改业务代码、不修改 ORM 模型定义的前提下，实现同一套代码在不同环境下自动适配不同的数据库驱动（SQLite, MySQL, PostgreSQL）。

## 决策

采用 **基于环境变量 `DATABASE_URL` 的 URL 前缀自动探测机制**，在应用启动时动态构建对应数据库驱动的 Engine。

### 核心实现逻辑

1. **驱动探测 (`_detect_driver`)**：在 `app/database.py` 中，解析 `DATABASE_URL` 字符串的前缀来判断目标驱动：
   - `sqlite:///` -> `sqlite`
   - `mysql://` 或 `mariadb://` -> `mysql`
   - `postgres://` 或 `postgresql://` -> `postgresql`
2. **Engine 构建 (`_build_engine`)**：
   - 针对 **SQLite**：采用 `connect_args={"check_same_thread": False}` 配置，适配 FastAPI 的多线程 ASGI 运行模式，避免 `ProgrammingError: SQLite objects created in a thread can only be used in that same thread` 错误。
   - 针对 **MySQL/PostgreSQL**：启用 SQLAlchemy 连接池 (`pool_size`, `max_overflow`)，并开启 `pool_pre_ping=True`，在连接失效时自动重连，增强网络稳定性。
3. **业务代码解耦**：所有业务代码仅依赖 `app.database` 模块暴露的 `engine` 实例和 `get_db` 依赖，**严禁**在业务层代码中硬编码数据库方言（Dialect）特定的 SQL 语法。

## 理由

1. **零侵入性**：通过解析 `DATABASE_URL` 这一标准 SQLAlchemy 配置项来判断驱动，开发人员无需修改代码，只需切换环境变量即可完成数据库切换，符合 12-Factor App 原则。
2. **维护成本低**：将所有驱动相关的差异处理（如连接参数、线程模型）集中封装在 `database.py` 中，避免散落在各处的 `if driver == 'mysql': ... elif ...` 代码分支。
3. **开发效率**：本地开发使用 SQLite 极大地降低了环境搭建门槛，新人上手更快。
4. **稳定性**：生产环境强制使用带连接池和 `pre_ping` 机制的配置，确保在高并发下的数据库连接稳定性。

## 后果

**正面**:
- 环境切换成本降至最低，只需一行 `export DATABASE_URL=...`。
- 架构清晰，驱动适配逻辑内聚在基础设施层。
- 极大提升了 CI/CD 的灵活性，可以在测试阶段使用 SQLite 快速运行单元测试，在集成测试阶段使用真实数据库。

**负面**:
- **ORM 功能子集受限**：为了保证 SQLite、MySQL 和 PostgreSQL 三者兼容，模型定义（`app/models/`）不能使用 PostgreSQL 特有的高级数据类型（如 `ARRAY`, `JSONB`），也不能使用 SQLite 不支持的 `AUTOINCREMENT` 特性，需使用 SQLAlchemy 通用类型（如 `String`, `Text`, `Integer`）。
- **SQL 方言限制**：若业务逻辑中需要书写原生 SQL（`text()`），必须使用 SQL 标准语法，不能使用如 MySQL 的 `LIMIT` 简写或 PostgreSQL 的 `ILIKE` 等方言特性。

## 替代方案

| 方案 | 否决理由 |
|------|---------|
| **多 Engine 实例（同时加载所有驱动）** | 资源浪费严重。在生产环境同时加载 SQLite 和 MySQL 引擎毫无意义；且增加了配置和维护的复杂度，容易引发实例间数据不一致。 |
| **基于配置文件（YAML/JSON）切换** | 增加了配置文件解析逻辑，不如直接利用环境变量 URL 前缀直观。配置文件管理和环境变量管理的本质是一致的。 |
| **手动指定驱动类型** | 违反自动化原则，需要开发者在部署时显式指定驱动，容易出错。URL 前缀是 SQLAlchemy 原生支持的事实标准。 |
