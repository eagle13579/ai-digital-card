# SLA 定义文档

> **文档版本**: v1.0  
> **最后更新**: 2026-07-01  
> **适用范围**: AI 数字名片 API 服务 (ncard-api)  
> **SLA 周期**: 月度统计

---

## 1. API 可用性 SLO

### 1.1 分级可用性目标

| 服务等级 | 可用性 SLO | 月度最大不可用时间 | 适用场景 |
|----------|-----------|-------------------|----------|
| **Tier 1** (核心读写 API) | **99.9%** | ≤ 43 分钟 50 秒 | 名片创建/更新/查询、用户认证、支付 |
| **Tier 2** (非核心 API) | **99.5%** | ≤ 3 小时 39 分钟 | 统计分析、历史记录、批量导出 |
| **Tier 3** (后台/管理 API) | **99.0%** | ≤ 7 小时 18 分钟 | 管理员操作、数据维护、内部工具 |

### 1.2 可用性计算方式

```
月度可用性 (%) = (总分钟数 - 不可用分钟数) / 总分钟数 × 100%
```

- **可用**：API 端点返回 HTTP 200–499 范围内的响应（不包含 503）。
- **不可用**：5xx 错误数超过总请求数的 5%；或完全无法建连（超时 > 10s）。
- **计划内维护**：提前 48 小时公告的维护窗口不计入不可用时间。
- **排除项**：因用户侧网络问题、DDoS 攻击、第三方依赖故障（如微信支付网关宕机）引起的不可用不计入。

### 1.3 可用性削峰策略

当可用性降至 SLO 阈值 0.1% 以内时，系统自动触发 **降级模式**：

- 读请求从数据库降级至 Redis 缓存（若缓存命中率 > 80%）
- 非核心功能（AI 推荐、个性化）返回默认值
- 写请求进入异步队列，返回 `202 Accepted` 而不是同步等待

## 2. 延迟 SLO

### 2.1 延迟分位数目标

| 分位数 | 目标值 | 测量方式 | 说明 |
|--------|--------|----------|------|
| **P50** (中位数) | **< 200 ms** | 滑动窗口 5 分钟 | 典型用户体验感受 |
| **P95** (第 95 百分位) | **< 1 s** | 滑动窗口 5 分钟 | 绝大多数用户的上限体验 |
| **P99** (第 99 百分位) | **< 3 s** | 滑动窗口 5 分钟 | 长尾优化基准 |

### 2.2 分端点延迟预算

| 端点分类 | P50 | P95 | P99 | 说明 |
|----------|-----|-----|-----|------|
| **读取类** (名片查询、搜索) | < 100 ms | < 500 ms | < 1.5 s | 缓存友好 |
| **写入类** (名片创建/更新) | < 300 ms | < 1 s | < 3 s | 含 DB 写入 |
| **AI 推理类** (智能推荐、摘要) | < 2 s | < 5 s | < 10 s | 依赖 LLM |
| **支付类** (发起支付、回调) | < 500 ms | < 1.5 s | < 3 s | 含第三方调用 |

### 2.3 延迟测量指标

使用 `ncard_http_request_duration_seconds` 直方图（`app/middleware/metrics.py`）采集：

| 指标名 | 类型 | 标签 |
|--------|------|------|
| `ncard_http_request_duration_seconds` | Histogram (10 buckets: 5ms–5s) | 无标签（全局） |
| `ncard_db_query_duration_seconds` | Histogram | — |
| `ncard_ai_inference_duration_seconds` | Histogram (wider buckets: 50ms–30s) | — |

## 3. 错误率 SLO

| 错误类型 | SLO | 例外 |
|----------|-----|------|
| **5xx 服务器错误** | < 0.1% 总请求数 | 计划维护、第三方依赖故障除外 |
| **4xx 客户端错误** | 无 SLO（由用户行为决定） | 429 限流除外 |
| **支付失败率** | < 1% | 用户余额不足/银行卡错误除外 |

### 错误率计算

```
错误率 (%) = 5xx 响应数 / 总请求数 × 100%
```

使用 `ncard_http_requests_total{status="5xx"}` 计数器。

## 4. SLA 违反与补偿

### 4.1 违反判定

在月度 SLO 统计周期结束后，若某项指标连续 **5 分钟滑动窗口** 超出 SLO 阈值，则记为一个 **SLI 违规事件**。当月 **累计违规时长** 用于判断是否触发补偿。

| 违反场景 | 判定条件 |
|----------|----------|
| 可用性违反 | 月度实际可用性低于 Tier 对应的 SLO |
| 延迟违反 | P95 延迟 > 1 s 持续 ≥ 5 分钟，月度累计 ≥ 30 分钟 |
| 错误率违反 | 5xx 错误率 ≥ 0.1% 持续 ≥ 5 分钟 |

### 4.2 补偿条款

| 月度可用性 | 补偿额度 | 补偿形式 |
|------------|----------|----------|
| < 99.9% 但 ≥ 99.5% | **10%** 月度费用抵扣 | 服务额度 |
| < 99.5% 但 ≥ 99.0% | **25%** 月度费用抵扣 | 服务额度 |
| < 99.0% 但 ≥ 98.0% | **50%** 月度费用抵扣 | 服务额度 |
| < 98.0% | **100%** 月度费用抵扣 | 服务额度 |

### 4.3 补偿流程

1. 系统自动检测 SLA 违反并生成报告（每月 5 日前完成上月统计）
2. 客户可通过 `/api/admin/sla-compensation` 端点查询违规记录
3. 补偿自动应用到下一期账单
4. 单次补偿上限为 **1 个月服务费**

### 4.4 补偿限制

- 不可抗力、计划维护、客户自身行为导致的不可用 **不在补偿范围内**
- Beta/预览版功能 **不适用** SLA 补偿
- 月度补偿总额不超过 **该月服务费的 100%**

## 5. Error Budget（错误预算）

### 5.1 预算计算

```
Error Budget (月度) = (1 - SLO) × 总请求数
```

示例（Tier 1, 99.9% SLO, 月均 1M 请求）：
```
Error Budget = (1 - 0.999) × 1,000,000 = 1,000 次错误
```

### 5.2 预算消耗

当月度累计 5xx 错误消耗到预算的以下阈值时触发不同等级响应：

| 消耗比例 | 响应动作 |
|----------|----------|
| **≤ 50%** | 正常操作，持续观察 |
| **50% – 75%** | 告警通知，优先修复高错误率路由 |
| **75% – 100%** | 停止非关键发布，冻结新功能上线 |
| **> 100%** | 启动紧急响应，回滚最近变更 |
