## 三、问题清单 (按严重程度排列)

### 🚨 P0 — 断裂路径 (用户流程中断)

| # | 问题 | 影响 |
|---|------|------|
| P0-1 | **首页「供需大厅」入口跳转 `/pages/market/index` → 404** | 页面不存在，用户点进去白屏 |
| P0-2 | **供需大厅「发布供需」按钮跳转 `/pages/post-need/index` → 404** | 用户无法发布供需信息 |
| P0-3 | **供需列表点击卡片跳转 `/pages/supply-demand/detail/index` → 404** | 用户看不到详细资料 |
| P0-4 | **搜索页100% Demo数据** × 7条硬编码结果 | 用户搜索得不到真实结果，搜索等于没用 |
| P0-5 | **首页AI推荐调用 `/api/v1/ai/recommend/hybrid/${cardId}` 接口可能不存在** | 后端没有该路由，API返回404 |

### 🔶 P1 — 功能缺失

| # | 问题 | 影响 |
|---|------|------|
| P1-1 | **无供需后端模型/路由** | 前端供需数据全靠Mock |
| P1-2 | **無「我的连接/联系人」页面** | 匹配建立连接后，用户在「我的」页面看不到已连接的人 |
| P1-3 | **无匹配确认交互流程** | match_records.state 支持 matched/pending/confirmed，但用户无法在 UI 上确认匹配 |
| P1-4 | **无消息通知机制** | 有人对你感兴趣、想连接你，用户收不到推送 |
| P1-5 | **无连接拒绝/解除功能** | 已有连接无法管理 |

### 🔶 P2 — 用户体验缺陷

| # | 问题 | 影响 |
|---|------|------|
| P2-1 | **供需大厅不在TabBar** | 用户需从首页点击入口，路径深 |
| P2-2 | **匹配推荐在「我的」页面用数字显示，无操作引导** | 用户看到「匹配数: 0」但不知去哪里操作 |
| P2-3 | **脱敏策略太激进** | free 用户看其他人名字首字+**，公司名前2字+**，无法判断是否值得联系 |
| P2-4 | **匹配分数展示混乱** | 前端×20/后端原始分/百分制混用，用户无法理解 |
| P2-5 | **无标签编辑引导** | 刚注册用户不知道要打标签才能做匹配 |

### 🔷 P3 — 技术债务

| # | 问题 | 说明 |
|---|------|------|
| P3-1 | **API版本混乱** | 前端用 `/api/v1/matches`，后端只有 `/api/match/*` |
| P3-2 | **供需前端组件无后端支撑** | 纯前端Demo结构，数据流未打通 |
| P3-3 | **匹配引擎性能隐患** | get_daily_recommendations 遍历所有用户 O(n²) |
| P3-4 | **匹配模型v2已训练但未集成** | matching_model_v2.pt 存在但线上用的是标签匹配 |

## 四、最佳实践方案

### Phase 1: 修复断裂路径 (紧急, 1天)

```
1. 首页入口: /pages/market/index → /pages/supply-demand/index
2. 后端新建供需模型 + CRUD API (SupplyDemand 表)
3. 新建供需发布页 /pages/supply-demand/publish
4. 新建供需详情页 /pages/supply-demand/detail
5. 搜索页接入真实API (调用 match/hybrid_search)
```

### Phase 2: 打通商务对接全流程 (3天)

```
完整的"找资源→匹配→连接→沟通"闭环:

1. 发现资源
   ├── 供需大厅 (浏览所有供需)
   ├── AI推荐 (基于标签+语义的个性化推荐)
   ├── 搜索 (按关键词搜索用户/供需/名片)
   └── 浏览他人名片

2. 表达兴趣 (触发匹配)
   ├── 点击「感兴趣」→ 创建 match_record
   ├── 双向匹配时 → 自动建立 connection
   └── 发送站内消息

3. 连接管理
   ├── 我的连接 (已建立的关系列表)
   ├── 待处理 (别人对我感兴趣)
   └── 已发送 (我感兴趣的)

4. 解锁联系方式
   ├── free: 每月3次解锁
   ├── vip: 每月10次
   └── pro: 不限次
```

### Phase 3: 资源平台升级 (5天)

```
1. 供需信息结构化
   ├── 分类: 技术/产品/渠道/资金/人才/场地
   ├── 发布模板: 标题+描述+标签+联系方式+预算范围
   ├── 支持附件/图片上传
   └── 状态管理: 进行中/已对接/已完成

2. 匹配增强
   ├── 标签权重动态调整 (用户行为反馈)
   ├── 语义搜索接入真实Embedding
   ├── 高级筛选: 行业/地区/规模/匹配度
   └── 匹配理由可解释 (为什么推荐这个人)

3. 商机转化跟踪
   ├── 对接记录 (谁联系了谁、什么时间)
   ├── 合作标签 (供应商/客户/伙伴/投资人)
   ├── 意向强度 (从浏览→感兴趣→连接→见面)
   └── 数据看板 (匹配数/连接数/合作数/转化率)
```

## 六、问题清单 (可直接提问)

以下是为团队准备的 **30个关键问题**，在开发资源平台时必须确认：

### 产品设计

1. 供需信息有哪些分类？(技术/产品/渠道/资金/人才/场地)
2. 发布供需的必填字段是什么？(标题+描述+标签+联系方式？)
3. 供需信息有生命周期吗？(发布→对接中→已完成→关闭)
4. 每人可发布几条供需？上限多少？
5. 供需可以编辑/删除吗？编辑后是否通知感兴趣的人？
6. AI推荐展示几条？什么排序？用户能否刷新？
7. 匹配理由怎么展示才易懂？(示例: "TA需要AI技术，你提供AI技术")
8. 连接后双方可以做什么？(发消息/交换电话/留语音)
9. Free会员每天/月能看到几个人的联系方式？
10. 脱敏策略是否应该调整为：free看名字全名但不显示电话？而非名字打星

### 技术实现

11. 后端供需模型需要哪些字段？(title/desc/category/tags/contact/status/user_id)
12. `/api/v1/matches` 的查询参数是什么？(page/purpose/user_id)
13. 语义搜索调用哪个embedding模型？本地还是远程？
14. 匹配引擎是否需要做增量计算？还是每次全量跑？
15. 搜索页接入真实API后，前端缓存策略是什么？
16. 站内消息用 WebSocket 还是轮询？
17. 连接建立后是否需要给双方都发消息通知？
18. 图片上传用本地存储还是OSS？
19. 匹配记录是否需要做分页？当前前端做的是全量返回
20. 连接数据(connections)是否需要做双向冗余？现在是双向两条记录

### 数据/增长

21. 当前Demo数据(张明远/李思琪等)什么时候替换为真实数据？
22. 如何引导新用户完善标签？注册流程中插入标签选择？
23. 匹配到连接的转化率1.4%，目标提升到多少？10%？
24. 是否需要在「我的」页面显示待处理的匹配请求数？
25. 访客量25次极低，是否需要主动push通知？
26. 如何衡量资源平台的成功？(匹配量/连接量/合作量/用户留存)
27. 是否需要配合会员权益来限制匹配/解锁次数？
28. 是否需要接入微信客服或企微来进行人工撮合？
29. 是否开放给非注册用户浏览部分供需信息？
30. 资源平台上线后，每天监控哪些核心指标？
