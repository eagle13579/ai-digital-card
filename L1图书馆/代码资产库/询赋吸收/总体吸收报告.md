## 四、吸收价值评估

### 必须保留（架构级设计）

| 设计 | 说明 |
|------|------|
| **双行关系记录** | TA的 connections 表双行设计是最大亮点。所有社交查询退化到最简单的 `WHERE user_id = ?`，无需 OR/UNION/DISTINCT，对小程序前端也一样友好 |
| **4级可见性矩阵** | 商务数据隐私的核心。特别是 `platform` 和 `network` 两级，精确控制了"同组织可见"和"好友可见" |
| **{code,message,data}响应格式** | 这个小规范能避免无数前后端联调问题。统一后 Mock/测试/文档 全部标准化 |
| **一键登录+自动注册** | 商务场景的"手机号即身份"模型，体验极佳 |

### 需改造适配

| 改造点 | 方向 |
|--------|------|
| Express + JWT → 微信小程序登录 | 使用 `wx.login()` 获取 code → 后端换取 openid/unionid → 签发自定义 Token |
| localStorage → wx Storage | `wx.setStorageSync` / `wx.getStorageSync` |
| Axios → wx.request 封装 | 封装统一的请求函数，保留 interceptors 逻辑 |
| Zustand → 小程序状态管理 | 使用 mobx-miniprogram 或自定义 Store 类 |
| SQLite → MySQL/云数据库 | 小程序后端建议使用 MySQL 或微信云开发数据库 |
| H5路由 → 小程序页面路由 | wx.navigateTo / wx.switchTab 替代 hash 路由 |

### 可忽略或低优先级

| 功能 | 说明 |
|------|------|
| 名片 OCR 上传 (upload.ts) | 小程序有原生 OCR 能力或可直接用微信的图片识别 |
| AI Chat 路由 (ai.ts) | 询赋的 AI 对话逻辑与数智名片的 AI 能力有重叠，需重新设计 |
| 反馈收集 | 非核心功能，可后期补 |

## 五、安装与实现建议

### 推荐实施路线

```
Phase 1 (核心基础设施) — 2-3天
  ├── F01: 微信小程序登录体系 + JWT Token
  ├── F05: connections 表双行设计
  └── F06: wx.request 统一封装 + 响应格式规范

Phase 2 (业务能力) — 3-4天
  ├── F04: resources 表 + 4级可见性过滤
  └── F03: BFS 社交图谱路径查找 + 建联请求审核

Phase 3 (组织能力) — 2-3天
  ├── F02: 平台/企业/圈子 + 角色体系 + 商业报告
  └── F07: 扫码名片交换

Phase 4 (AI增强) — 持续迭代
  └── 接入 AI 能力（名片分析、关系推荐、智能搜索）
```

### 数据库选型建议

| 询赋 (SQLite) | AI数智名片推荐 | 理由 |
|---------------|---------------|------|
| 单文件数据库 | MySQL / 微信云开发数据库 | 多用户并发、云端部署、数据安全 |
| uuid() 主键 | 自增 ID 或 snowflake ID | 索引性能更好，排序友好 |
| 本地文件存储 | 腾讯云 COS / 微信云存储 | 名片图片的持久化、CDN 加速 |

## 七、最终结论

**询赋项目对AI数智名片有极高的吸收价值。**

其核心架构（双行关系记录 + 4级可见性矩阵 + BFS社交图谱 + 一键登录 + 统一响应格式）构成了一个完整的**商务社交后端范式**。虽然前端框架不同（H5+Axios+Zustand vs 微信小程序），但后端的心智模型、架构决策和数据库设计可以**几乎无损地迁移**。

最大的单一代价是前端框架切换（H5→小程序），但这属于实现层面的替换，不影响架构设计本身。**建议优先吸收 F05(双向关系记录)、F04(可见性矩阵)、F01(一键登录) 三个核心 Feature**，它们构成了 AI数智名片的"商务社交身份系统"骨架。
