# F05 — 双向关系记录

## 心智模型（Mental Model）

社交关系是**双向的**，但数据库记录通常是单向的。询赋使用 **双行记录模式**：每次建立连接时，同时写入正向（A→B）和反向（B→A）两条记录。

这不是冗余设计，而是为了让查询始终以 `WHERE user_id = ?` 的方式工作，无需 `OR contact_id = ?` 的双向索引，大幅简化SQL。

```
connections 表数据示例：

| id | user_id | contact_id | status   |
|----|---------|------------|----------|
| 1  | A       | B          | approved |  ← 正向：A的好友是B
| 2  | B       | A          | approved |  ← 反向：B的好友是A

查询A的好友: WHERE user_id = 'A' AND status = 'approved' → 得到B
查询B的好友: WHERE user_id = 'B' AND status = 'approved' → 得到A
```

## Architecture Decision Records

### ADR-012: 使用双行记录而非单行 + OR 查询

| 项目 | 决策 |
|------|------|
| 决策 | 每次建联操作为每条连接写入2行（正向+反向） |
| 替代方案 | 单行记录 + `WHERE user_id = ? OR contact_id = ?` |
| 理由 | 1) 查询全走 `user_id` 索引，无需 OR 索引合并；2) 排序、分页、count 都简单直接；3) 后续的关系强度、标签等字段可各自独立更新 |
| 代价 | 存储翻倍；写入时需事务保证两条记录一致性 |
| 结论 | 读多写少的社交场景，双行记录的读写比为优化收益远超存储成本 |

### ADR-013: 审核操作同步更新两条记录

| 项目 | 决策 |
|------|------|
| 上下文 | 接收方审核一条请求后，正向反向状态需保持一致 |
| 决策 | reviewConnection() 中通过反向记录的 `user_id` 定位正向记录同步更新 |
| 理由 | 事务一致性；避免一条 approved 另一条仍 pending |
| 后果 | 需通过两条 SQL 更新（一条查一条改），非原子操作但无竞争条件风险 |

## 核心代码提取

```typescript
// 建联请求 — 双行写入
static requestConnection(userId: string, targetUserId: string, message?: string) {
  // ...校验已存在关系...

  const id = uuid();
  db.prepare(
    'INSERT INTO connections (id, user_id, contact_id, source, status) VALUES (?, ?, ?, ?, ?)'
  ).run(id, userId, targetUserId, 'platform', 'pending');

  // 反向记录
  const reverseId = uuid();
  db.prepare(
    'INSERT INTO connections (id, user_id, contact_id, source, status) VALUES (?, ?, ?, ?, ?)'
  ).run(reverseId, targetUserId, userId, 'platform', 'pending');

  return { id, status: 'pending' };
}

// 审核请求 — 同步更新双行
static reviewConnection(connectionId: string, userId: string, approved: boolean) {
  const conn = db.prepare(
    'SELECT * FROM connections WHERE id = ? AND contact_id = ?'
  ).get(connectionId, userId) as any;

  const newStatus = approved ? 'approved' : 'rejected';
  db.prepare('UPDATE connections SET status = ?, updated_at = unixepoch() WHERE id = ?')
    .run(newStatus, connectionId);

  // 同步更新反向记录
  db.prepare(
    'UPDATE connections SET status = ?, updated_at = unixepoch() WHERE user_id = ? AND contact_id = ?'
  ).run(newStatus, userId, conn.user_id);
}
```

## 查询对比

| 场景 | 单行模式 | 双行模式（询赋） |
|------|---------|----------------|
| 查询A的好友 | `WHERE (user_id='A' OR contact_id='A') AND status='approved'` | `WHERE user_id='A' AND status='approved'` |
| 复杂索引 | 需复合索引或 UNION | 单字段 `user_id` 索引即可 |
| 分页查询 | DISTINCT 去重复杂 | 直接 COUNT + LIMIT |
| 关系强度 per-user | 无法区分A→B和B→A强度 | 各自独立存储 |

## 数据库 Schema

```sql
CREATE TABLE connections (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL REFERENCES users(id),   -- 视角用户
  contact_id TEXT NOT NULL REFERENCES users(id), -- 关系对象
  source TEXT DEFAULT 'platform',                -- 关系来源
  status TEXT CHECK(status IN ('pending', 'approved', 'rejected')),
  strength REAL DEFAULT 0,                       -- A 对 B 的关系强度评分
  label TEXT DEFAULT '',                         -- A 给 B 打的标签（可选扩展）
  created_at DEFAULT unixepoch(),
  updated_at DEFAULT unixepoch(),
  UNIQUE(user_id, contact_id)                    -- 确保单向唯一
);
```
