# F04 — 4级可见性矩阵

## 心智模型（Mental Model）

每一份资源（名片、商机、需求等）都有**4级可见性**，决定谁能看到它。这是一个从"公开"到"私密"的同心圆模型：

```
                    ┌──────────────────┐
                    │  private         │ 只有自己可见
                    │  ┌────────────┐  │
                    │  │ network    │  │  好友可见
                    │  │ ┌────────┐ │  │
                    │  │ │platform│ │  │  同平台成员可见
                    │  │ │ ┌────┐ │ │  │
                    │  │ │ │public│ │  │  所有人可见
                    │  │ │ └────┘ │ │  │
                    │  │ └────────┘ │  │
                    │  └────────────┘  │
                    └──────────────────┘
```

**可见性等级**（从宽到窄）：
1. `public` — 所有用户（含未登录）可见
2. `platform` — 同平台的成员可见
3. `network` — 好友（connections中 status='approved'）可见
4. `private` — 仅自己可见

## Architecture Decision Records

### ADR-010: 可见性在 SQL 层面过滤，而非应用层后过滤

| 项目 | 决策 |
|------|------|
| 上下文 | 搜索资源时需要根据当前用户返回可见的子集 |
| 决策 | 在 SQL WHERE 子句中使用复合 OR 条件直接过滤 |
| 理由 | 减少数据传输量；利用数据库索引；代码集中一处 |
| 后果 | SQL 复杂度增加；需传递当前用户ID作为参数 |

### ADR-011: 四级可见性使用字符串枚举存储

| 项目 | 决策 |
|------|------|
| 上下文 | 可见性标记需要清晰、可扩展 |
| 决策 | resources.visibility 字段存储 'public'/'platform'/'network'/'private' |
| 理由 | 字符串比位掩码更可读、易调试；不需要位运算 |
| 后果 | 索引效率略低于整型枚举，但在百万级数据内无差异 |

## 核心代码提取

```typescript
// 可见性过滤 SQL 条件
conditions.push(`
  (r.visibility = 'public' 
   OR r.owner_id = ?                                  -- 自己的资源
   OR (r.visibility = 'platform' AND r.platform_id IN (
       SELECT platform_id FROM platform_members WHERE user_id = ?
     ))                                                -- 同平台成员的资源
   OR (r.visibility = 'network' AND (
       r.owner_id IN (
         SELECT contact_id FROM connections WHERE user_id = ? AND status = 'approved'
       )
       OR r.owner_id IN (
         SELECT user_id FROM connections WHERE contact_id = ? AND status = 'approved'
       )
     )))                                               -- 好友的资源
`);
values.push(params.userId, params.userId, params.userId, params.userId);
```

## 数据库 Schema 映射

```sql
CREATE TABLE resources (
  id TEXT PRIMARY KEY,
  type TEXT NOT NULL,              -- 资源类型: 'card'/'opportunity'/'demand'/'knowledge'
  title TEXT NOT NULL,
  owner_id TEXT REFERENCES users(id),
  platform_id TEXT REFERENCES platforms(id),
  description TEXT DEFAULT '',
  industry TEXT DEFAULT '',
  city TEXT DEFAULT '',
  visibility TEXT DEFAULT 'network' CHECK(visibility IN ('public','platform','network','private')),
  metadata TEXT,                   -- JSON 扩展字段
  created_at DEFAULT unixepoch(),
  updated_at DEFAULT unixepoch()
);
```

## 吸收建议

| 询赋实现 | AI数智名片适配方案 |
|----------|-------------------|
| SQLite子查询 | MySQL/MariaDB 下同样适用 |
| 默认 network | 建议默认设为 public 增加曝光，或允许用户配置默认值 |
| 资源类型 type | 可扩展为 'card'/'product'/'service'/'project' |
| 搜索+可见性一体 | 必须保留同一架构，这是核心能力 |
