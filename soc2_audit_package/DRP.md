## 1. 恢复目标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **RTO** (恢复时间目标) | **≤ 4 小时** | 从灾难发生到服务完全恢复的最大允许时间 |
| **RPO** (恢复点目标) | **≤ 1 小时** | 允许丢失的最大数据量（时间跨度） |
| **MTD** (最大容忍停机) | **≤ 8 小时** | 业务可接受的最长停机时间 |
| **MTO** (最大容忍中断) | **≤ 30 分钟** | 部分降级服务可接受的短暂中断 |
| **SLA 合规** | **99.9%** | 年度可用性承诺 |

### 1.1 服务优先级

| 优先级 | 服务 | RTO | RPO | 说明 |
|--------|------|-----|-----|------|
| **P0** | 用户认证 / API 网关 | 30 分钟 | 实时 | 核心入口，中断即影响所有用户 |
| **P1** | 名片管理 / 交易服务 | 1 小时 | ≤ 5 分钟 | 核心业务功能 |
| **P2** | AI 服务 / 分析报表 | 2 小时 | 1 小时 | 增值功能，可暂时降级 |
| **P3** | CRM / 集成服务 | 4 小时 | 1 小时 | 非关键路径 |

## 2. 灾难分类

### 2.1 灾难等级定义

| 等级 | 定义 | 示例 | 响应时限 |
|------|------|------|----------|
| **Tier 1** (严重) | 核心服务完全不可用，数据丢失 | 服务器崩溃、数据库损坏、勒索软件 | 15 分钟内响应 |
| **Tier 2** (重大) | 主要功能降级或部分可用 | 网络攻击、第三方依赖故障 | 30 分钟内响应 |
| **Tier 3** (轻微) | 非核心功能受影响 | 配置错误、证书过期 | 1 小时内响应 |

### 2.2 灾难类型

#### 2.2.1 服务器崩溃 (Tier 1)

**可能原因**:
- 硬件故障（磁盘、内存、电源）
- 操作系统内核 Panic
- 云实例意外终止
- Docker 容器 OOM

**影响**: 核心服务不可用，可能导致数据不一致

**前置信号**:
- CPU/内存持续高水位（>90% × 5 分钟）
- 磁盘 I/O 等待时间飙升
- 系统日志频繁 OOM Killer

#### 2.2.2 数据损坏 (Tier 1~2)

**可能原因**:
- 数据库存储引擎故障
- 软件 Bug 导致写入错误数据
- 意外 DROP/UPDATE/DELETE 操作
- 存储层比特翻转/静默数据损坏

**影响**: 业务数据丢失或错误，严重可致全量数据不可用

**前置信号**:
- 数据库一致性检查失败
- 应用层报数据校验错误
- 查询返回异常结果

#### 2.2.3 网络攻击 (Tier 1~2)

**可能原因**:
- DDoS 攻击
- 勒索软件加密
- SQL 注入/SSRF 利用
- 凭据泄露导致的未授权访问

**影响**: 服务不可用、数据泄露、数据被加密

**前置信号**:
- 流量异常陡增
- 认证失败率飙升
- 安全设备告警（WAF/IDS）

#### 2.2.4 第三方依赖故障 (Tier 2~3)

**可能原因**:
- 云服务商（AWS/Azure/阿里云）可用区故障
- CDN/PaaS/SaaS 商服务中断
- 上游 API 限流或下线
- DNS 解析失败

**影响**: 依赖该服务的功能降级或不可用

**前置信号**:
- 第三方健康检查告警
- API 响应超时率上升
- DNS 解析延迟增加

## 3. 备份策略

### 3.1 数据库备份

| 备份类型 | 频率 | 保留周期 | 存储位置 | 说明 |
|----------|------|----------|----------|------|
| 全量备份 | 每日 01:00 UTC | 30 天 | 主存储 + 异地对象存储 | PostgreSQL pg_dump / MySQL mysqldump |
| WAL 归档 | 连续 | 7 天 | 异地对象存储 | 用于 Point-in-Time Recovery (PITR) |
| 增量备份 | 每 6 小时 | 14 天 | 异地对象存储 | 减少全量备份间隔 |

**验证**: 每日备份后自动执行 `pg_restore --test` 或等价校验

### 3.2 代码与配置备份

| 备份项 | 方式 | 频率 | 说明 |
|--------|------|------|------|
| 源代码 | Git (GitHub) | 每次提交 | 含所有分支，远程仓库容灾 |
| 环境配置 | 加密存档 | 每次变更 | `.env.production` + 密钥 |
| Docker 镜像 | Container Registry | 每次构建 | 带标签保留最近 30 个版本 |
| 基础设施代码 | Terraform/K8s YAML | 每次变更 | 版本化管理 |

### 3.3 备份存储架构

```
┌─────────────────────────────────────────────────────┐
│                    生产环境                           │
│  ┌──────────┐    ┌──────────┐    ┌──────────────┐   │
│  │ 主数据库  │    │ 应用服务器 │    │ 对象存储(本地) │   │
│  └────┬─────┘    └──────────┘    └──────┬───────┘   │
│       │ 每日备份                          │           │
│       ▼                                  ▼           │
│  ┌──────────┐                    ┌──────────────┐   │
│  │ 备份服务  │───────────────────▶│  备份暂存区   │   │
│  └──────────┘                    └──────┬───────┘   │
└──────────────────────────────────────────┼──────────┘
                                           │ 同步
                                           ▼
┌──────────────────────────────────────────────────────┐
│                    异地灾备中心                        │
│  ┌──────────────────────────────────────────────┐    │
│  │          对象存储 (异地 / 冷存储)              │    │
│  │  ├─ backups/db/full/       (30天)            │    │
│  │  ├─ backups/db/wal/        (7天)             │    │
│  │  ├─ backups/env/          (加密)             │    │
│  │  └─ backups/code/         (Git bundle)       │    │
│  └──────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────┘
```

### 3.4 备份加密

- 所有异地备份使用 **AES-256-GCM** 加密
- 加密密钥由密钥管理服务 (KMS/Vault) 保管
- 备份恢复需提供解密密钥（运维负责人 + 安全负责人双人授权）

## 4. 恢复流程

### 4.1 恢复阶段概览

```
识别灾难 → 评估影响 → 通知团队 → 执行恢复 → 验证恢复 → 复盘总结
   ┌───┐      ┌───┐      ┌───┐      ┌───┐      ┌───┐      ┌───┐
   │ 1 │─────▶│ 2 │─────▶│ 3 │─────▶│ 4 │─────▶│ 5 │─────▶│ 6 │
   └───┘      └───┘      └───┘      └───┘      └───┘      └───┘
```

### 4.2 阶段 1: 识别灾难

**触发条件**:
- 监控系统告警（Prometheus + Alertmanager）
- 用户反馈（客服 / 社交媒体）
- 定期健康检查失败
- 安全事件告警（WAF/IDS/SIEM）

**操作步骤**:
1. 确认告警真实性，排除误报
2. 初步判断灾难类型和等级
3. 在事故管理系统中创建事件 (Incident Ticket)

**输出**: 事件编号 + 初步灾难评级

### 4.3 阶段 2: 评估影响

**操作步骤**:
1. 确定受影响的系统和服务范围
2. 评估受影响用户数量及业务影响
3. 判断是否需要立即升级灾难等级
4. 估算恢复所需时间

**工具**:
- 监控仪表盘 (Grafana)
- 日志系统 (Loki / ELK)
- 应用性能监控 (APM)

**输出**: 影响范围报告 + 资源需求清单

### 4.4 阶段 3: 通知团队

**通知层级**:

| 角色 | 姓名 | 联系方式 | 通知渠道 |
|------|------|----------|----------|
| 运维负责人 | (待填写) | (待填写) | 电话 + Slack |
| 安全负责人 | (待填写) | (待填写) | 电话 + Slack |


### 4.5 阶段 4: 执行恢复

#### 4.5.1 服务器崩溃恢复

```bash
# 1. 从灾备节点启动
kubectl scale deployment app --replicas=3 -n production

# 2. 检查数据库连接
pg_isready -h $DB_HOST -p $DB_PORT

# 3. 验证应用健康
curl -f http://localhost:8000/health

# 4. 逐步恢复流量
# 蓝绿部署模式下切换 DNS/负载均衡
```

#### 4.5.2 数据损坏恢复

```bash
# 1. 停服或切换至只读模式
kubectl scale deployment app --replicas=0 -n production

# 2. 从最近备份恢复数据库
# 选择恢复方式：
#   a) 点时间恢复 (PITR) — 丢失最少数据
#   b) 全量备份恢复 — 最快但可能丢失更多数据

# 3. 验证数据完整性
pg_checksums -r /var/lib/postgresql/data

# 4. 重新启动服务
# 5. 验证业务逻辑
```

#### 4.5.3 网络攻击恢复

```bash
# 1. 隔离受影响系统（从网络层面断开）
# 2. 切换至备用 IP / CDN 清洗
# 3. 如果数据被加密：
#   a) 从异地备份恢复数据
#   b) 重建实例（不信任原实例）
# 4. 通知安全团队取证
# 5. 更改所有受影响凭据
```

#### 4.5.4 第三方依赖恢复

```bash
# 1. 启用服务降级模式（功能降级而非全量不可用）
# 2. 尝试切换至备用提供商
# 3. 如果 DNS 故障，切换至备用 DNS
# 4. 等待上游恢复或启用本地缓存回退
```

### 4.6 阶段 5: 验证恢复

**验证清单**:

| 检查项 | 方法 | 预期结果 |
|--------|------|----------|
| API 健康检查 | `curl /health` | 返回 200 + `{"status":"ok"}` |
| 数据库连接 | `pg_isready` | 返回 `accepting connections` |
| 用户认证 | 测试登录流程 | 正常登录 |
| 核心业务流程 | 创建/编辑/删除名片 | 功能正常 |
| 数据完整性 | 比较备份与恢复后关键数据 | 一致 |
| 监控告警 | 检查告警是否恢复 | 无 P0/P1 告警 |
| 安全扫描 | 快速漏洞扫描 | 无新增高危漏洞 |

### 4.7 阶段 6: 复盘总结

**复盘会议 (Incident Postmortem)**:
- 时间：恢复后 48 小时内
- 参与人：所有响应人员 + 相关方
- 议程：
  1. 时间线回顾
  2. 根因分析 (5 Whys / RCA)
  3. 做得好的地方
  4. 待改进之处
  5. 行动计划及负责人

**输出**:
- 事故报告 (Incident Report)
- 改进项跟踪 (Action Items)

## 6. 演练计划

### 6.1 演练频率

| 演练类型 | 频率 | 说明 |
|----------|------|------|
| 桌面推演 (Tabletop) | 每季度 1 次 | 讨论式演练，不涉及实际操作 |
| 技术演练 | 每半年 1 次 | 实际操作备份恢复流程 |
| 全量演练 | 每年 1 次 | 模拟真实灾难场景，全流程演练 |

### 6.2 演练场景

| 场景 | 类型 | 目标 | 验收标准 |
|------|------|------|----------|
| 数据库意外 DROP | 技术演练 | 从备份恢复，RTO ≤ 2h | 数据完全恢复，服务正常 |
| 服务器宕机 | 技术演练 | 从灾备启动，RTO ≤ 30min | 服务自动恢复 |
| DDoS 攻击模拟 | 桌面推演 | 验证响应流程 | 15 分钟内识别并启动防护 |
| 勒索软件加密 | 全量演练 | 从异地备份恢复，RTO ≤ 4h | 全部服务恢复，数据零丢失 |

### 6.3 演练记录

每次演练后需记录：
- 演练时间、参与人员
- 实际 RTO / RPO 与目标的偏差
- 发现的问题及改进项
- 下次演练计划调整
