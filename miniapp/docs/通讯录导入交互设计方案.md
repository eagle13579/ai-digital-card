# 链客宝微信小程序「通讯录导入」功能 — 交互设计方案

> **版本**: v1.0  
> **日期**: 2026-07-14  
> **状态**: 待评审  
> **关联页面**: `pages/network/import/`（现有导入页面扩展）

---

## 1. 用户流程

### 1.1 主流程

```
打开小程序 → 进入「人脉」Tab → 点击「导入联系人」按钮
    ↓
进入导入页面（现有 pages/network/import/index）
    ↓
选择「通讯录导入」方式
    ↓
调用 wx.chooseContact（微信通讯录选择器）
    ↓
用户勾选一个或多个联系人 → 点「确定」
    ↓
显示「导入进度」页（loading + 进度条 + 计数）
    ↓
已选的联系人逐条上传至后端匹配
    ↓
匹配完成 → 显示「导入完成」页（匹配结果预览）
    ↓
用户可「邀请好友」或「返回人脉」
```

### 1.2 分支流程

#### A. 拒绝授权（微信通讯录权限）

```
wx.chooseContact 调用 → 微信弹出授权弹窗
    ↓
用户点「拒绝」
    ↓
微信弹窗关闭，触发 fail 回调（errMsg 含 "cancel"）
    ↓
页面显示 Toast：「已取消通讯录导入」
    ↓
页面回退到「导入方式选择」界面
    ↓
用户可选择其他导入方式（手动输入/扫码/邀请）
```

#### B. 已导入过（重复检测）

```
用户选择「通讯录导入」
    ↓
wx.chooseContact 成功返回联系人列表
    ↓
前端先根据 phone/wechatId 与本地已存 contacts 做碰撞检测
    ↓
有重复 → 弹 Modal 提示：
    「已有 X 位联系人已导入，是否继续导入其余 Y 位？」
    ↓
用户点「跳过重复」→ 只导入新联系人
    ↓
用户点「全部重新导入」→ 覆盖旧数据
```

#### C. 导入失败

```
后端匹配接口返回错误 / 网络中断 / 超时
    ↓
已成功匹配的 N 条不撤回（部分成功语义）
    ↓
失败的联系人进入「导入失败」列表
    ↓
页面底栏显示：
    「已导入 M 位 | N 位导入失败，点击重试」
    ↓
用户点击「重试」→ 重新发送失败项
```

#### D. 取消导入（导入进行中）

```
导入进度页显示中 → 用户点击「取消导入」
    ↓
Modal 确认弹窗：「确认取消导入？已导入的联系人不会丢失」
    ↓
用户选「确认取消」→ 停止后续上传 → 显示已导入结果
    ↓
用户选「继续导入」→ 关闭弹窗，继续
```

## 3. 交互细节

### 3.1 授权失败的处理

| 场景 | 用户操作 | APP 表现 | 用户感知 |
|------|---------|---------|---------|
| 首次拒绝 | 点「拒绝」 | `fail` 回调被触发，errMsg 含 "cancel" | Toast「已取消授权」→ 回退到方式选择 |
| 拒绝后再次点击 | 再次点「通讯录导入」 | 微信弹窗不再弹出，直接 `fail` | Toast「请在系统设置中开启通讯录权限」→ 引导至设置 |
| 系统权限关闭 | 设置中关闭了通讯录权限 | `wx.chooseContact` 始终 `fail` | 引导页显示「去设置」按钮 → 调用 `wx.openSetting` |
| 基础库版本低 | 用户微信版本低 | API 不存在 → catch 处理 | Toast「当前微信版本不支持，请升级后使用」 |

```javascript
// 授权失败的统一处理策略
function handleChooseContactFail(err) {
  const errMsg = err.errMsg || ''

  if (errMsg.includes('cancel')) {
    // 用户手动取消——友好提示，不弹错误
    wx.showToast({ title: '已取消通讯录导入', icon: 'none' })
  } else if (errMsg.includes('auth deny') || errMsg.includes('not authorized')) {
    // 授权被拒绝——引导去设置页
    wx.showModal({
      title: '需要通讯录权限',
      content: '请在系统设置中允许链客宝访问通讯录',
      confirmText: '去设置',
      success: (res) => {
        if (res.confirm) wx.openSetting()
      },
    })
  } else {
    // 其他异常（基础库版本、系统错误等）
    wx.showToast({ title: '通讯录导入暂不可用', icon: 'none' })
  }
}
```

### 3.2 重复导入检测

**检测逻辑**（前端执行，避免不必要的网络请求）：

```
对 wx.chooseContact 返回的联系人列表：
  forEach contact:
    用 contact.phoneNumber 去重：
      - 若 phoneNumber 已存在于 this.data.contacts（本地缓存）
      - 或 phoneNumber 已存在于本批次中
      → 标记为「重复」

  去重结果：
    - 全部重复 → 弹窗「所选联系人已全部导入」
    - 部分重复 → 弹窗「X 位已导入，是否跳过？」
    - 无重复 → 直接进入导入流程
```

**重复判定优先级**：
1. `phoneNumber` 完全匹配（最高优先级）
2. `wechatId` 完全匹配（次优先级）
3. `name` 模糊匹配（最低优先级，仅提示不自动跳过）

### 3.3 导入中的错误处理

```
┌─────────────────────────────────────────────────┐
│ 逐条上传匹配                                     │
│                                                  │
│  ┌──── 成功 ────→ 计数 +1, 继续下一条           │
│  │                                               │
│  第N条 ──── 失败（HTTP错误） ─→ 标记失败，继续   │
│  │          （网络超时）   ─→ 重试3次 → 标记失败 │
│  │          （后端code≠0） ─→ 记录错误原因      │
│  └──── 全部完成 ──→ 跳转完成页                  │
│                                                  │
│  进度更新：每处理 1~3 条更新一次进度              │
│  （避免高频 setData 导致卡顿）                    │
└─────────────────────────────────────────────────┘
```

**失败重试策略**：
- 单条失败：最多重试 **3 次**，间隔 500ms
- 重试仍失败：标记为「失败」，记录错误原因（后端返回的 message）
- 整批失败（网络断开）：暂停处理，弹出重试/取消对话框

### 3.4 取消导入的反馈

```
用户点击「取消导入」
    ↓
wx.showModal({
  title: '确认取消？',
  content: '已导入的联系人不会丢失，未完成的将不保存',
  confirmText: '确认取消',
  cancelText: '继续导入',
})
    ↓
确认取消 → 
  - 标记 cancelFlag = true
  - 正在处理的当前条继续完成
  - 不再发起新的请求
  - 剩余未处理的联系人写入「待导入」队列（供下次恢复）
    ↓
跳转到完成页（只显示已完成的匹配结果）
```

## 4. 前端 API 调用

### 4.1 wx.chooseContact 调用方式

```javascript
// 调用微信通讯录选择器
wx.chooseContact({
  success: (res) => {
    // res 结构：
    // {
    //   name: '张三',           // 联系人姓名
    //   phoneNumber: '13800138000',  // 手机号
    //   wechatId: 'zhangsan_wx',     // 微信号（可能为空）
    // }
    // 注意：wx.chooseContact 只能单选！每次只返回一个联系人
    // 需要多次调用来支持多选，或者改用 wx.chooseContact 的扩展版本
    // 微信基础库 2.27.0+ 支持 multiple 参数
  },
  fail: (err) => {
    handleChooseContactFail(err)
  },
  complete: () => {
    // 无论成功失败都会执行
  },
})
```

> ⚠️ **重要说明**：`wx.chooseContact` 在微信基础库中为单次单选接口。要实现多选，有两种方案：
>
> **方案 A（推荐）**：循环调用 `wx.chooseContact`，每次选择一位用户，把已选的传回作为排除列表。用户选完一位后自动弹出下一次选择，直到用户主动退出。
>
> **方案 B**：使用 `wx.chooseContact` 的 `multiple: true` 参数（需基础库 2.27.0+）。如果支持，可一次选择多人。
>
> **降级策略**：若基础库不支持多选，自动降级为方案 A（循环调用）。

### 4.2 选择后的数据处理

```javascript
// 从 wx.chooseContact 获取的联系人 → 前端预处理
function prepareContactsForImport(contactList) {
  return contactList
    .filter(c => c.phoneNumber && /^1[3-9]\d{9}$/.test(c.phoneNumber))
    // 仅保留有有效手机号的联系人
    .map(c => ({
      name: c.name || '未知',
      phone: c.phoneNumber,
      wechatId: c.wechatId || '',
      source: 'contacts',  // 标识来源为通讯录导入
      raw: c,              // 保留原始数据供后续使用
    }))
}
```

## 5. 错误状态与异常处理

### 5.1 网络异常

| 阶段 | 错误表现 | 处理方式 | 用户感知 |
|------|---------|---------|---------|
| 调用 `wx.chooseContact` 前 | 无网络 | 检测 `wx.getNetworkType`，无网络时弹窗提示 | 「当前无网络，请连接后重试」+ 去设置开关 |
| 上传匹配中 | 网络中断 | 暂停处理，记录已完成的进度 | Toast「网络已断开」→ 底部显示「恢复后继续」按钮 |
| 上传匹配中 | 弱网（3G/慢速） | 放超时时间为 10s，超时重试 3 次 | 进度条停在当前位置，不报错，等待重试 |
| 上传匹配后 | 网络恢复 | 自动检测网络状态变化（`wx.onNetworkStatusChange`） | 自动继续未完成的上传 |

**自动恢复逻辑**：

```javascript
// 监听网络变化
wx.onNetworkStatusChange((res) => {
  if (res.isConnected && this.data.importingFailed) {
    // 网络恢复 & 有失败项 → 弹窗询问是否继续
    wx.showModal({
      title: '网络已恢复',
      content: `还有 ${failedCount} 位联系人未匹配，是否继续？`,
      confirmText: '继续',
      cancelText: '稍后',
      success: (modal) => {
        if (modal.confirm) this.retryFailedImport()
      },
    })
  }
})
```

### 5.2 授权拒绝

| 场景 | 处理 | UI 反馈 |
|------|------|---------|
| 点「拒绝」按钮 | 记录 `_authDenied = true`，下次点击优先检测 | Toast + 回到方式选择 |
| 第二次点击（已拒绝过） | 检测到 `_authDenied = true`，跳过 `wx.chooseContact` 直接引导 | 弹窗引导至系统设置 |
| 从系统设置返回 | `wx.onAppShow` 监听 | 自动重新尝试 `wx.chooseContact`（若已授权则继续流程） |

### 5.3 后端超时

| 超时类型 | 阈值 | 处理 |
|---------|------|------|
| 批量接口超时 | 15s（与 `request.js` 的 `REQUEST_TIMEOUT` 一致） | 降级为逐条上传 |
| 单条接口超时 | 5s | 重试 3 次，每次间隔 1s，失败则标记 |
| 整批长时间无响应 | 30s 无任何成功 | 弹窗「匹配服务繁忙，请稍后重试」，提供重试按钮 |

**超时降级流程**：

```
批量接口 POST /api/contacts/match
    ↓ 超时（15s）
降级为逐条上传 POST /api/contacts/match-one
    ↓ 单条超时（5s）
重试 3 次 → 仍超时 → 标记为「失败」
    ↓
所有单条处理完毕（成功+失败）→ 跳转到完成页
    ↓
失败项提供「重试全部」按钮 → 重新走批量接口
```

### 5.4 数据异常

| 异常类型 | 检测方式 | 处理 |
|---------|---------|------|
| 返回数据格式异常 | 后端返回的 data 非数组或缺少关键字段 | 标记该条为「数据异常」，跳过继续，完成页显示该错误 |
| 联系人号码无效 | 前端预验证 / 后端返回 `invalid_phone` | 失败列表显示「号码无效」 |
| 返回的 user 信息为空 | `matched` 中的 user 对象为 `null` 或缺少 `id` | 视为匹配失败，标记原因「数据异常」 |
| 重复匹配 | 后端返回的 matched 列表中有 `id` 相同 | 前端去重，保留第一个匹配结果 |
| 手机号格式异常 | 非 11 位 / 含特殊字符 | 前端预过滤，标记为「号码格式异常」，不入库不请求 |

### 5.5 异常处理汇总

```
┌───────────────────────────────────────────────────────┐
│                  异常处理决策树                        │
├───────────────────────────────────────────────────────┤
│                                                        │
│  wx.chooseContact 调用                                 │
│    ├── success → 预处理联系人数据                      │
│    │   ├── 有效联系人 > 0 → 进入导入流程              │
│    │   └── 有效联系人 = 0 → Toast「未选择联系人」     │
│    └── fail → 错误分支处理                             │
│        ├── cancel → Toast「已取消」→ 回退              │
│        ├── auth deny → 引导到系统设置                   │
│        └── 其他 → Toast「暂不可用」→ 回退              │
│                                                        │
│  导入匹配流程                                          │
│    ├── 成功 → 计数 +1                                  │
│    ├── 单条失败 → 重试 3 次 → 标记 → 继续             │
│    ├── 网络中断 → 暂停 → 恢复后继续                    │
│    ├── 批量超时 → 降级逐条 → 逐条超时 → 标记 → 继续  │
│    └── 用户取消 → 停止 → 显示已完成的                  │
│                                                        │
│  导入完成页                                            │
│    ├── 全部成功 → 「导入完成」+ 匹配列表               │
│    ├── 部分失败 → 成功列表 + 失败列表 + 重试按钮       │
│    └── 全部失败 → 「导入失败」+ 错误原因 + 重试按钮    │
│                                                        │
└───────────────────────────────────────────────────────┘
```

## 6. 性能与体验优化建议

### 6.1 批量处理

- 单次最多导入 **50** 位联系人（微信选择器上限）
- 超过 50 位：提示「一次最多选择 50 位，请分批导入」
- 前端每 **5 条** 更新一次进度（减少 `setData` 频率）
- 批量接口请求：50 条/次

### 6.2 缓存策略

- 已导入的联系人本地缓存（`wx.setStorageSync`），key: `imported_contacts`
- 匹配失败的记录本地缓存（`failed_imports`），供后续「重试」使用
- 缓存过期时间：7 天（清理过期缓存）

### 6.3 加载状态

- 导入进度页显示骨架屏（Skeleton）加载态
- 完成页使用懒加载（虚拟列表），避免长列表卡顿
- 导入过程中保持屏幕常亮

### 6.4 用户引导

- 首次使用「通讯录导入」时显示功能引导浮层（3 步提示）
- 引导浮层仅在首次展示，用户勾选「不再提示」后不再弹出
- 匹配失败的联系人提供「一键邀请」入口，方便用户手动邀请

## 7. 微信限制与兼容性

| 限制项 | 说明 | 应对 |
|-------|------|------|
| `wx.chooseContact` 基础库版本 | 最低支持 2.11.0 | 版本检测，不支持时隐藏该入口 |
| 多选支持 | `multiple` 参数需基础库 2.27.0+ | 低版本降级为循环单选 |
| 单次选择上限 | 微信限制最多 50 位 | 超量提示分批导入 |
| 返回数据类型 | 仅返回 `name`, `phoneNumber`, `wechatId`，无头像 | 前端使用首字生成占位头像 |
| 授权状态 | 无法主动检查授权状态 | 通过 fail 回调反向推断 |
| 调用频率 | `wx.chooseContact` 无频率限制（文档未标注） | 正常使用无需限制 |
