## §3 五层架构设计

```
┌──────────────────────────────────────────────────────────┐
│                   用户触达层                               │
│  ┌──────────────────┐      ┌────────────────────────┐    │
│  │  微信小程序        │      │  网页版 SPA            │    │
│  │  (WXML+JS+WXSS)   │ ◄──► │  (Alpine.js 6页面)    │    │
│  │  15页面→20页面     │      │  + 分享链接 + 二维码   │    │
│  └────────┬─────────┘      └───────────┬────────────┘    │
│           │  wx.request / api.js        │ fetch / axios   │
│           ▼                             ▼                 │
│  ┌────────────────────────────────────────────────────┐   │
│  │         API网关层 (go-aiport.com/lkapi)             │   │
│  │  统一认证 · 限流 · trace_id · 版本管理 /api/v1/     │   │
│  └────────────────────┬───────────────────────────────┘   │
│                       ▼                                   │
│  ┌────────────────────────────────────────────────────┐   │
│  │          业务服务层 (digital_brochure_api.py)       │   │
│  │  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐    │   │
│  │  │画册  │ │认证  │ │匹配  │ │信任  │ │供需  │    │   │
│  │  │CRUD  │ │系统  │ │引擎  │ │网络  │ │发布  │    │   │
│  │  └──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘    │   │
│  │     └────────┴────────┴────────┴────────┘         │   │
│  └────────────────────┬───────────────────────────────┘   │
│                       ▼                                   │
│  ┌────────────────────────────────────────────────────┐   │
│  │          共享数据层 (SQLite 7表)                     │   │
│  │  小程序 · 网页版 · 所有客户端共享同一数据库          │   │
│  └────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────┘
```

**核心原则**: 小程序和网页版共享同一套 API + 数据库，不做数据隔离。小程序专注「移动端画册浏览/分享/扫码」，网页版专注「编辑/管理/分析」。

## §5 数据打通方案

### 共享API路径（小程序和网页版共用）

| 用途 | API路径 | 小程序调用 |
|:-----|:--------|:-----------|
| 画册CRUD | `/api/v1/brochures` | `api.request('/api/v1/brochures')` |
| 画册详情 | `/api/v1/brochures/{user_id}` | `api.request('/api/v1/brochures/' + userId)` |
| 信任网络 | `/api/v1/brochures/{user_id}/trust_network` | `api.request(...)` |
| 匹配计算 | `/api/v1/match` | `api.request('/api/v1/match', 'POST', {user_id})` |
| 扫码跳转 | `/scan/{brochure_id}` | 微信扫描二维码 → H5页面 |
| 访客记录 | `/api/v1/brochure/{id}/visitors` | `api.request(...)` |

### 身份打通

```
网页版注册(手机+密码) ←→ 小程序登录(手机+验证码+微信openid)
                              ↓
                     共享 auth_users 表
                   user_id 作为画册唯一标识
```

### 二维码打通

```
小程序生成二维码 → 保存/分享 → 微信扫描 → webview加载H5画册页 → 记录访客
                                                    ↓
                                             匹配引擎触发
```

## §6 最佳实践铁律

### 铁律1：API先行，页面其次
小程序和网页版共用同一套 `/api/v1/*` 端点，不单独为小程序写后端逻辑。所有新功能先加API，再同时实现小程序和网页版前端。

### 铁律2：数据源唯一
数据库只维护一份。小程序和网页版的数据100%互通。用户在小程序编辑画册 → 网页版即时可见。

### 铁律3：小程序专注「浏览+分享」
小程序定位：扫码打开、微信分享、快速浏览。画册编辑器的复杂UI保留在网页版（或作为P1优化）。

### 铁律4：离线兜底
小程序画册数据缓存本地(`wx.setStorageSync`)，无网络时仍可查看已缓存的画册。

### 铁律5：二维码即入口
每个画册一个二维码，微信扫描直接打开小程序画册页。二维码 = 数字名片。
