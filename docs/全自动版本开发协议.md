## 一、分支体系

```
master  ← 生产环境 ← 你说"确认"我才合并
  ↑
releaseV1.0  ← 发布候选版本（当前版本）
  ↑
develop  ← 大模块开发集成
  ├── feature/XXX  ← 小模块/小修复（从 develop 开出，改完合并回 develop）
  └── feature/XXX
```

**规则：**
- `master` — 只进不出。**你说"确认"我才合并**，合并后打 tag
- `releaseV1.0` — 当前版本的发布分支，从 develop 分出，稳定后等确认上 master
- `develop` — 日常开发集成分支。所有 feature 最终合到这里
- `feature/XXX` — 小模块/小修复，从 develop 开出，改完合回 develop

## 四、自动闭环节点

```
你说"修复XX"
  ↓
我:
  git checkout develop && git pull
  git checkout -b feature/fix-XX
  改代码 → 验证语法
  git add -A
  git commit -m "fix: 修复XX问题"
  git push origin feature/fix-XX
  ↓
"改好了，分支 feature/fix-XX 已推送，等确认后合并到 develop"
```

---

## 五、确认上线流程

```
你说"确认发布"
  ↓
我:
  Step 1: feature 分支 → 合并到 develop
    git checkout develop
    git merge feature/XXX
    git push origin develop

  Step 2: develop → 合并到 releaseV1.0
    git checkout releaseV1.0
    git merge develop
    git push origin releaseV1.0

  Step 3:（等你再次"确认"）releaseV1.0 → 合并到 master
    git checkout master
    git merge releaseV1.0
    git tag v1.X.X
    git push origin master --tags
  ↓
"已上线 v1.X.X"
```

**注意：** 只有合并到 master 才算上线。我说"改好了"时代码在 feature 或 develop 上，**只有你说"确认"才会碰 master**。

## 七、铁律

1. **改完自动推 feature 分支** — 不等你提醒
2. **不确认不上 master** — 你说"确认"前 master 不动
3. **commit 写中文** — 你看得懂为主
4. **只改 miniapp/** — 不碰 backend/frontend 等其他目录
5. **不推送 = 没做完** — 我说"改好了"时 GitHub 上一定有代码
