## 一、三档任务自动路由

| 你说 | 我判断 | Git动作 |
|:-----|:-------|:--------|
| "改个bug" / "修复XX" | 代码量<50行，单文件 | 直接在 dev-miniapp 改 → commit `fix: ...` → push |
| "加个XX功能" / "新增XX" | 新文件或>50行改动 | 直接在 dev-miniapp 改 → commit `feat: ...` → push |
| "开发XX大功能" | 跨目录改动，影响>5文件 | 创建分支 `feat/功能名` → 开发 → PR → 合并 dev-miniapp |

## 二、commit 信息规范（自动生成，你不需要管）

```
类型: 简短描述 (中文，不超过20字)
```

| 类型 | 说明 | commit示例 |
|:-----|:-----|:-----------|
| `fix:` | 细节错误修复 | `fix: 名片编辑页头像加载失败` |
| `feat:` | 新增小模块/功能 | `feat: 供需大厅发布按钮` |
| `refactor:` | 重构代码（不改功能） | `refactor: 抽离api请求层` |
| `style:` | 样式/UI调整 | `style: 首页卡片圆角统一` |
| `docs:` | 文档修改 | `docs: 更新开发最佳实践` |

## 四、版本迭代规则

```
master ← 里程碑版本，由 dev-miniapp 合并
  ↑ PR合并 + 打 tag
dev-miniapp ← 日常所有开发都在这里
  ↑ 所有 feat / fix 直接提交
```

**打 tag 时机**（我会问你的）:
- 一个完整功能开发完毕
- 你准备部署前
- 格式: `v1.0.0` → `v1.1.0` → `v1.1.1`

## 六、三个你常问的场景

**Q: 今天改了一堆小bug，怎么提交？**
→ 我按改动内容自动归组，同类型的合并一个 commit

**Q: 改到一半想看看之前的版本？**
→ `git stash` 存当前改动，切 master 看完，切回来 `git stash pop`

**Q: 想把 dev-miniapp 合并到 master？**
→ 你告诉我"发版"，我自动创建 PR → 合并 → 打 tag → push master

## 七、铁律（我违反了你直接指出来）

1. **不说"提交"我也提** — 每次改完自动 commit+push，不等你提醒
2. **commit 信息必须中文** — 不用英文，你看得懂为主
3. **不改非小程序目录** — 只动 `miniapp/` 下文件
4. **不推送 = 没做完** — 我说"改好了"时，代码一定已在 GitHub 上
5. **dev-miniapp 永远可跑** — 每次提交前验证语法，不让坏代码上去
