# 服务等级协议 (SLA)

> **项目**: AI 数智名片 (AI Digital Business Card)
> **版本**: v1.0
> **生效日期**: 2026-07-01
> **适用范围**: 后端 API 服务 (`backend`) 及 AI 服务 (`ai_service`)
> **SLA 等级**: **99.9%** (月度可用性)

---

## 1. 服务可用性承诺

| 服务组件 | 月度可用性 | 年度可用性 | 允许停机时间（月度） |
|----------|------------|------------|---------------------|
| 后端 API (`:8201`) | >= 99.9% | >= 99.8% | <= 43.2 分钟 |
| AI 服务 (`:8202`)  | >= 99.5% | >= 99.0% | <= 3.6 小时 |
| 前端静态资源 (Nginx) | >= 99.9% | >= 99.8% | <= 43.2 分钟 |
| **整体平台** | **>= 99.9%** | **>= 99.8%** | **<= 43.2 分钟** |

### 1.1 可用性计算公式

```
月度可用性 (%) = (总分钟数 - 不可用分钟数) / 总分钟数 x 100%

其中:
  - 不可用分钟数 = 用户无法访问核心功能的总时长
  - 排除计划内维护窗口（见第 6 节）
```

## 2. 响应时间承诺

| 端点分类 | P50 (中位数) | P95 (百分位) | P99 (百分位) |
|----------|:------------:|:------------:|:------------:|
| **静态资源** (HTML/CSS/JS) | <= 100ms | <= 300ms | <= 500ms |
| **API -- 轻量查询** (名片读取、搜索) | <= 200ms | <= 500ms | <= 1s |
| **API -- 写入** (创建、更新名片) | <= 500ms | <= 1s | <= 3s |
| **API -- AI 流式** (RAG 对话) | 首字 <= 1s | 首字 <= 2s | 首字 <= 5s |
| **API -- 认证** (登录、注册) | <= 500ms | <= 1s | <= 2s |
| **文件上传** (<= 10MB) | <= 2s | <= 5s | <= 10s |

### 2.1 测量方法

- 响应时间在服务端 `otel` 中间件中统计（不包含客户端网络延迟）
- 每秒采样，按 60 秒窗口聚合计算百分位
- 排除速率限制 (HTTP 429) 和客户端错误的请求 (HTTP 4xx)

## 3. 故障分类与响应时间

### 3.1 故障等级定义

| 等级 | 标签 | 定义 | 示例 |
|------|------|------|------|
| **P0 -- 严重** | Critical | 核心功能完全不可用，影响所有用户 | API 全量不可用、数据库宕机、认证系统崩溃 |
| **P1 -- 高** | High | 核心功能部分不可用，影响大量用户 | AI 服务不可用、文件上传失败、支付链路中断 |
| **P2 -- 中** | Medium | 非核心功能不可用，或影响有限用户 | 向量搜索回退到 TF-IDF、消息推送延迟、前端样式错乱 |
| **P3 -- 低** | Low | 轻微问题，无用户影响 | 审计日志延迟、非关键告警、文档错误 |

### 3.2 响应与恢复时间

| 等级 | 响应时间 (A) | 恢复时间 (RTO) | 更新频率 |
|------|:----------:|:--------------:|:--------:|
| **P0 -- 严重** | <= 15 分钟 | <= 1 小时 | 每 30 分钟 |
| **P1 -- 高**  | <= 30 分钟 | <= 4 小时 | 每 1 小时 |
| **P2 -- 中**  | <= 2 小时 | <= 24 小时 | 每 4 小时 |
| **P3 -- 低**  | 下一个工作日 | <= 72 小时 | 每日 |

> **A -- 响应时间**: 从告警触发或用户报告到工程师确认并开始排查的时间
> **RTO -- 恢复时间目标**: 从告警触发到服务恢复正常的最长时间

### 3.3 告警通知渠道

| 等级 | 通知方式 | 通知对象 |
|------|----------|----------|
| P0 | 电话 + 短信 + 即时通讯 (企业微信/钉钉) | 值班工程师 + 技术负责人 |
| P1 | 即时通讯 + 短信 | 值班工程师 |
| P2 | 即时通讯 | 相关团队 |
| P3 | 邮件 / 工作项 | 团队负责人 |

## 4. 赔偿条款

### 4.1 月度可用性赔偿

| 月度可用性 | 服务额度赔偿 | 赔偿形式 |
|------------|:----------:|----------|
| 99.0% ~ 99.9% (未达标) | **10%** 月度服务费 | 抵扣下月账单 |
| 95.0% ~ 99.0% | **25%** 月度服务费 | 抵扣下月账单 |
| < 95.0% | **50%** 月度服务费 | 抵扣下月账单 |

### 4.2 赔偿申请流程

1. 客户在故障发生后 **30 天内** 提交赔偿申请至 `support@liankebao.top`
2. 申请需包含：故障时间、影响描述、工单号或监控截图
3. 我方在 **5 个工作日内** 核实并答复
4. 获批准的赔偿在 **下个账单周期** 自动抵扣

### 4.3 赔偿限制

- 单次故障赔偿上限为 **月度服务费的 50%**
- 累计赔偿不超过 **年度服务费的 25%**
- 以下情况 **不在赔偿范围**:
  - 计划内维护（提前 48 小时通知）
  - 第三方服务故障 (DeepSeek API、微信/支付宝支付网关、云厂商)
  - 用户侧原因 (网络问题、错误配置、超出速率限制)
  - 不可抗力 (自然灾害、战争、政府行为)
  - 免费 / 试用用户

## 5. 例外情况

以下情况的停机不计入可用性计算:

| 类别 | 说明 |
|------|------|
| **计划内维护** | 提前 >= 48 小时在服务状态页公告; 每月 <= 2 次, 每次 <= 2 小时 |
| **第三方依赖故障** | DeepSeek API、微信支付、支付宝、云基础设施 |
| **用户滥用** | 因触发速率限制 (HTTP 429) 或 DDoS 导致的不可用 |
| **测试/预发布环境** | 仅生产环境 (`production`) 受 SLA 覆盖 |
| **API 版本弃用** | 旧版 API 在弃用通知 90 天后不再受 SLA 保障 |

## 7. 监控与报告

### 7.1 监控覆盖

| 监控项 | 工具 | 频率 |
|--------|------|------|
| API 可用性 | Uptime Kuma / 外部探针 | 每 1 分钟 |
| API 响应时间 | OpenTelemetry + Prometheus | 实时 |
| 错误率 | Sentry + Prometheus | 实时 |
| 资源使用 (CPU/内存/磁盘) | Prometheus + Node Exporter | 每 15 秒 |
| TLS 证书过期 | 自动脚本 | 每日 |

### 7.2 SLA 报告

- **月度 SLA 报告**: 每月 5 日前发布上月可用性报告
- **报告内容**: 当月可用性、故障事件列表、MTTR/MTBF 统计、改进措施
- **报告发送**: 邮件至客户注册邮箱及企业微信管理群

## 8. 定义

| 术语 | 定义 |
|------|------|
| **可用性** | 服务可正常访问和响应的时间百分比 |
| **停机时间** | 用户无法正常使用核心功能的时间段 |
| **MTTR** | Mean Time To Recovery -- 平均恢复时间 |
| **MTBF** | Mean Time Between Failures -- 平均故障间隔 |
| **RTO** | Recovery Time Objective -- 恢复时间目标 |
| **RPO** | Recovery Point Objective -- 数据恢复点目标 (<= 5 分钟) |
