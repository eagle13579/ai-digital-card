## 测试模块1：ai/match智能匹配（筛选+交换名片）

### 页面路径：pages/ai/match/index

#### 【P0】匹配筛选功能使用picker但未初始化selectedIndustryIndex/selectedRegionIndex
- **问题描述**: WXML中picker的value绑定为 `{{selectedIndustryIndex}}` 和 `{{selectedRegionIndex}}`，但data中未定义这两个字段，picker初始化时value为undefined，可能导致picker异常。
- **严重程度**: P0
- **复现步骤**: 1. 进入智能匹配页面 2. 点击行业picker下拉 3. picker的value未定义导致异常
- **涉及文件**: pages/ai/match/index.wxml:10,20

#### 【P0】"解锁"功能无实际操作
- **问题描述**: `unlock(e)`方法仅显示Toast"解锁功能开发中"，销售总监无法通过匹配列表交换名片或建立联系。
- **严重程度**: P0
- **复现步骤**: 1. 进入匹配页面 2. 看到一个匹配对象 3. 点击进入 4. 显示"解锁功能开发中"
- **涉及文件**: pages/ai/match/index.js:54-56

#### 【P1】匹配列表数据完全依赖Mock
- **问题描述**: 匹配页面无真实API对接，所有匹配推荐数据来自MockService.getMatchList()。标记为"92%"匹配度的数据实际是硬编码的测试数据。
- **严重程度**: P1
- **复现步骤**: 1. 进入匹配页面 2. 看到3个匹配对象 3. 他们的头像URL一致使用外部CDN
- **涉及文件**: pages/ai/match/index.js:20-21

#### 【P1】匹配列表中tags为空时不显示任何内容
- **问题描述**: match-tags区域使用`wx:for="{{item.tags}}"`，当tags不存在或为空数组时，该区域留白，破坏卡片布局。
- **严重程度**: P1
- **复现步骤**: 1. Mock返回的匹配对象tags字段为undefined 2. 页面不渲染tags区域 3. 卡片底部有空白

#### 【P2】筛选条件不够丰富
- **问题描述**: 仅支持行业和地域两个筛选维度，缺少匹配度范围、标签筛选、公司规模等业务关键筛选维度。
- **严重程度**: P2
- **复现步骤**: 1. 进入匹配页 2. 只有行业和地域两个picker 3. 无法按匹配度筛选

## 测试模块2：brochure/create画册创建（快速填写）

### 页面路径：pages/brochure/create/index

#### 【P0】图片上传使用本地临时路径，无法持久化
- **问题描述**: avatar、companyImages使用wx.chooseImage获取的tempFilePaths，这些路径是临时的，小程序重启后失效。没有上传到服务器或转存为本地持久文件。
- **严重程度**: P0
- **复现步骤**: 1. 创建画册并上传头像 2. 提交成功后预览 3. avatar使用tempFilePath 4. 关闭小程序重启后头像不可用
- **涉及文件**: pages/brochure/create/index.js:86-96

#### 【P1】表单验证的简介字数提示不够明显
- **问题描述**: validateForm()中建议bio至少50字，但仅通过Logger.warn记录日志，用户看不到任何提示。
- **严重程度**: P1
- **复现步骤**: 1. 填写简介少于50字 2. 提交表单 3. 验证通过（bio非必填）4. 控制台有警告但用户无感知

#### 【P1】技能标签输入使用`,`和`，`分割但不支持粘贴
- **问题描述**: onSkillTagsInput绑定到input事件，需要逐字符输入逗号分隔，不支持批量粘贴或退格删除后重新分割。
- **严重程度**: P1
- **复现步骤**: 1. 在技能标签输入框粘贴"A,B,C" 2. 分割为三个标签正确 3. 尝试退格删除一个字符 4. 整段重新解析可能不符合预期

#### 【P1】添加案例时没有案例数量限制提示
- **问题描述**: addCase可以无限添加案例，但画册页面模板只有无限滚动逻辑。案例过多时画册过长，影响移动端体验。
- **严重程度**: P1
- **复现步骤**: 1. 点击"添加案例" 2. 可以连续添加10+案例 3. 提交后画册页面数量巨大

#### 【P2】表单刷新会丢失已填数据
- **问题描述**: 画册创建表单没有自动保存草稿功能。用户填写大量信息后，如果切换页面或小程序被系统回收，所有已填数据丢失。
- **严重程度**: P2
- **复现步骤**: 1. 填写公司介绍、发展历程等长内容 2. 切换到微信聊天再回小程序 3. 表单数据丢失
- **建议**: 使用Storage作为草稿箱

#### 【P2】合作意向选择器可点区域过小
- **问题描述**: purposeOptions使用flex布局，点击标签仅触发该标签区域（约40px高度），容错率低。
- **严重程度**: P2
- **复现步骤**: 1. 选择"寻找投资" 2. 点击标签文字区域有效 3. 点击标签边缘无效

## 测试模块4：ai/scan扫码（协议解析）

### 页面路径：pages/ai/scan/index

#### 【P1】OCR名片识别使用Mock硬编码结果
- **问题描述**: scanImage()方法返回固定mock数据（姓名张三、公司星辰科技等），无论扫描什么名片都返回相同内容。
- **严重程度**: P1
- **复现步骤**: 1. 拍一张名片 2. 识别完成后一直显示"张三/产品经理/星辰科技有限公司" 3. 无真实OCR识别能力
- **涉及文件**: pages/ai/scan/index.js:110-122

#### 【P1】扫码次数限制无会员差异化
- **问题描述**: maxScans固定为3次，未根据用户memberLevel区分（Free 3次、Pro 100次）。
- **严重程度**: P1
- **复现步骤**: 1. 扫描3次名片 2. 第四次显示"扫描次数已用完" 3. 无法继续使用，即使是Pro/Enterprise用户
- **涉及文件**: pages/ai/scan/index.js:51

#### 【P1】扫码结果中"保存为名片"实际上并未保存到后端
- **问题描述**: saveCard()方法仅setTimeout后Toast提示并跳转到预览页，实质没有保存扫描结果到名片库后端。
- **严重程度**: P1
- **复现步骤**: 1. OCR识别出名片 2. 点击"保存为名片" 3. Toast"已保存到名片" 4. 跳转到预览页 5. 预览页展示的是Mock画册数据，非扫描结果

#### 【P2】二维码扫码协议解析后自动跳转但无加载状态
- **问题描述**: handleScanProtocol中setTimeout 800ms后跳转，此800ms内页面无任何加载状态显示，用户可能困惑。
- **严重程度**: P2
- **复现步骤**: 1. 扫码后 2. Toast显示"识别到user" 3. 等待800ms 4. 跳转 5. 800ms等待期内页面无进度指示
- **涉及文件**: pages/ai/scan/index.js:167-185

## 跨模块问题（适用于周总视角）

#### 【P0】完整名片交换闭环断裂
- **问题描述**: 销售总监的核心流程：扫码/匹配发现 → 查看对方名片 → 交换联系方式 → 建立信任关系 → 跟进客户。当前每个步骤均为孤立的Mock实现，无法形成业务闭环。
- **严重程度**: P0
- **涉及模块**: ai/match, ai/scan, qrcode, network/graph

#### 【P1】名片数据无任何加密/权限控制
- **问题描述**: 所有名片数据通过MockService返回完整信息（电话、微信等），无数据脱敏或权限控制逻辑。
- **严重程度**: P1
- **涉及模块**: 所有展示联系方式的页面
