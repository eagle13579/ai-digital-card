## 一、ai/index AI智能中心

### 测试操作流程
1. 从首页"AI智能"按钮进入 AI 中心（`/pages/ai/index/index`）
2. 查看8个功能入口的网格布局
3. 分别点击每个功能图标跳转
4. 测试未登录状态下的登录拦截

### 发现的问题

| # | 问题描述 | 严重程度 | 复现步骤 |
|---|---------|---------|---------|
| 1 | **`go()` 函数中 `store.checkLogin()` 仅弹Toast不阻止跳转** — `checkLogin()` 在未登录时通过 `wx.navigateTo({ url: '/pages/login/index' })` 跳转登录页的同时又执行了 `wx.navigateTo({ url })` 跳转到目标页，导致在功能页和登录页之间反复跳转 | **P1** | 未登录 → 进入AI中心 → 点击任一功能 → 观察页面跳转行为 |
| 2 | **`ai/gaia`、`ai/feedback`、`ai/config` 页面代码未读到** — 三个页面在 features 数组中有定义入口，但 `pages/ai/` 目录下未找到对应的 `gaia`、`feedback`、`abtest`、`modelserve` 目录，跳转会404 | **P1** | AI中心 → 点击"盖娅进化大脑"或"反馈建议"或"AI客服配置" |
| 3 | **AI中心仅为静态菜单页** — 缺少 AI 能力概览、使用统计、推荐功能等动态内容，对 CEO 来说信息密度不足 | **P2** | 进入AI中心 → 观察页面内容 |
| 4 | **未登录状态无引导提示** — 未登录用户进入AI中心时看不到任何限制提示，直接点击功能才会触发登录跳转，体验突兀 | **P2** | 未登录 → 进入AI中心 |

## 二、ai/chat AI智能对话

### 测试操作流程
1. 从AI中心进入AI对话（`/pages/ai/chat/index`）
2. 查看初始欢迎消息
3. 在输入框输入消息并发送
4. 查看AI回复
5. 测试未登录状态下的登录蒙层
6. 点击"立即登录"按钮

### 发现的问题

| # | 问题描述 | 严重程度 | 复现步骤 |
|---|---------|---------|---------|
| 1 | **聊天列表不会自动滚动到底部** — 新消息添加后，`scroll-view` 的 `scroll-into-view="bottom"` 只在初始化时生效，新增消息后未动态更新，用户需手动下滑查看最新消息 | **P1** | 发送多条消息 → 观察滚动条是否自动到底 |
| 2 | **AI回复为固定硬编码文本，非真实AI** — 无论用户发送什么内容，AI 总是回复同一段预设文本"你好！我是AI数智名片助手..."，缺乏实际对话能力 | **P2** | 输入任意消息 → 观察AI回复 |
| 3 | **聊天记录无持久化** — 页面 `onUnload` 后聊天记录全部丢失，刷新页面回到初始状态 | **P2** | 发送几条消息 → 返回再进入 → 聊天记录丢失 |
| 4 | **`getApp().isLoggedIn()` 函数不一定存在** — `onLoad` 和 `send()` 中调用 `app.isLoggedIn()`，但在 `app.js` 中不一定定义了该方法，可能导致报错 | **P2** | 检查 app.js 中是否定义了 isLoggedIn 方法 |
| 5 | **登录蒙层可能遮挡输入区域** — `login-mask` 使用固定定位覆盖整个页面，但输入区域的 `z-index` 可能高于蒙层，导致未登录用户仍可操作输入框 | **P2** | 未登录 → 进入AI对话 → 尝试点击输入框 |
| 6 | **发送按钮禁用状态视觉反馈弱** — `disabled="{{loading || !canSend}}"` 的按钮样式与普通按钮差异不够明显（无灰色样式等） | **P3** | 输入为空或正在加载时观察发送按钮 |

## 三、ai/generate AI内容生成

### 测试操作流程
1. 从AI中心进入内容生成（`/pages/ai/generate/index`）
2. 在5种内容类型标签间切换（自我介绍/个人口号/介绍信/求职简历/自荐信）
3. 切换3种子风格（专业版/轻松版/创意版等）
4. 填写表单：姓名、职位、公司、行业、关键词、经验年限、主要成就、核心技能
5. 点击"AI生成内容"按钮
6. 查看生成结果，点击"复制"和"应用到名片编辑"

### 发现的问题

| # | 问题描述 | 严重程度 | 复现步骤 |
|---|---------|---------|---------|
| 1 | **预填的个人信息未显示在输入框中** — `onLoad` 中从 MockService 获取了用户档案并设置 `inputData`，但所有 `<input>` 未绑定 `value="{{inputData.xxx}}"`，导致用户看到的输入框全是空的 | **P1** | 进入AI生成页 → 观察表单字段 → 姓名/职位/公司均为空 |
| 2 | **生成过程中无进度反馈** — 点击生成后 `loading` 设为 true，但UI上没有loading指示器或禁用按钮反馈，用户可能不确定是否正在生成 | **P2** | 点击"AI生成内容" → 观察按钮状态 |
| 3 | **内容类型切换时表单输入会被覆盖** — 切换 type 或 subtype 时只清空了 `generated` 和 `result`，但 `inputData` 保持不变，切换后再切回来输入数据还在，这可能或可能不是预期行为 | **P2** | 填写表单 → 切换到其他类型 → 再切回来 |
| 4 | **生成结果无字数/长度限制** — 表单字段无任何长度限制，生成的内容可能过长或过短，影响使用体验 | **P2** | 在关键词/成就中填入大量文本后生成 |
| 5 | **"应用到名片编辑"无实际数据传递** — 点击后只复制到剪贴板、跳转名片编辑页，但生成的内容不会自动填入编辑表单 | **P2** | 生成内容 → 点击"应用到名片编辑" |
| 6 | **"复制"和"应用到名片编辑"在生成后同时显示** — 两个按钮有功能重叠（复制到剪贴板），用户可能困惑两者的区别 | **P3** | 生成内容后观察两个按钮 |
| 7 | **生成结果使用纯文本渲染** — 对于含有 `\\n` 换行符的模板内容，在 `<text>` 标签中不会渲染为真实换行，而是显示为空格或 `\n` 字符 | **P2** | 生成自我介绍 → 观察结果中的换行 |
| 8 | **`skills` 字段在5种内容类型的部分模板中使用 `split(/[,，]/)`** — 若用户使用全角逗号，分割正常；若使用英文逗号后带空格（如 "Python, Java"），分割结果会含前导空格 | **P3** | 输入带空格的技能列表 → 生成简历/自荐信 |

## 四、ai/match智能匹配

### 测试操作流程
1. 从AI中心进入智能匹配（`/pages/ai/match/index`）
2. 查看默认匹配列表
3. 使用行业筛选器切换行业
4. 使用地域筛选器切换地域
5. 点击匹配列表中的用户
6. 观察筛选结果变化和空状态

### 发现的问题

| # | 问题描述 | 严重程度 | 复现步骤 |
|---|---------|---------|---------|
| 1 | **`selectedIndustryIndex` 和 `selectedRegionIndex` 未初始化** — 筛选器使用 `<picker>` 的 `value="{{selectedIndustryIndex}}"`，但 data 中只定义了 `selectedIndustry: '全部'` 和 `selectedRegion: '全部'`，未定义对应的 `selectedIndustryIndex` 和 `selectedRegionIndex`，默认选中项可能不正确 | **P1** | 进入匹配页 → 观察行业/地域筛选器的默认选中项 |
| 2 | **`onIndustryChange` 和 `onRegionChange` 后筛选器显示不同步** — 虽然 `selectedIndustry` 更新了，但 `selectedIndustryIndex` 未同步更新，picker 下次打开时可能停留在之前的选中位置 | **P1** | 切换行业筛选 → 再次打开 picker → 观察选中状态 |
| 3 | **所有匹配项点击均显示"解锁功能开发中"** — 匹配推荐的核心交互（查看详情、交换名片、建立联系）全部未实现 | **P2** | 点击任一匹配用户 |
| 4 | **缺少搜索关键词输入框** — 仅有行业和地域两个筛选维度，缺少按姓名/公司名称搜索的能力，筛选粒度不够 | **P2** | 观察匹配页的筛选区域 |
| 5 | **无"清除筛选"或"重置"按钮** — 筛选后用户需逐个将筛选条件改回"全部"，操作路径长 | **P2** | 设置筛选条件后尝试重置 |
| 6 | **匹配列表无分页/加载更多机制** — 一次性加载全部数据，列表较长时可能影响性能 | **P2** | 匹配数据较多的场景 |

## 五、ai/scan AI名片扫描

### 测试操作流程
1. 从AI中心进入名片扫描（`/pages/ai/scan/index`）
2. 点击"拍照/选择图片"进行 OCR 识别
3. 查看 OCR 识别结果
4. 编辑识别结果的字段
5. 点击"保存为名片"
6. 点击"重新扫描"
7. 使用"扫一扫"功能扫描二维码
8. 查看扫码结果和自动跳转
9. 测试扫描次数限制

### 发现的问题

| # | 问题描述 | 严重程度 | 复现步骤 |
|---|---------|---------|---------|
| 1 | **OCR 识别结果为完全固定的 Mock 数据** — 不管拍摄什么名片，识别结果始终是"张三 / 产品经理 / 星辰科技有限公司"，不具备真实识别能力 | **P2** | 拍摄任意名片 → 查看识别结果 |
| 2 | **`checkLoginAndLimit()` 中调用 `app.getState()` 但无此方法** — 代码使用 `app.getState()` 检查 token，但 `app.js` 中更可能通过 `store.getState()` 实现，导致可能报错或登录检查失效 | **P1** | 触发检查登录态的场景 |
| 3 | **`saveCard()` 使用硬编码延时模拟保存** — 无真实API调用，保存后跳转到 `/pages/brochure/preview/index`（不带id参数），可能导致预览页加载异常 | **P2** | OCR识别后 → 点击"保存为名片" |
| 4 | **扫描次数存储在 localStorage 但不支持跨设备同步** — 使用 `wx.getStorageSync('scanCount')` 管理，用户清除缓存或更换设备后次数重置 | **P2** | 扫描达到上限 → 清除微信缓存 → 重新进入 |
| 5 | **`scanQRCode()` 扫码成功后扫描次数+1，但无法识别自己的二维码** — 如果扫到自己名片的二维码，识别逻辑与OCR混用同一次数池，逻辑上应该分开 | **P2** | 扫码识别自己的名片二维码 |
| 6 | **OCR和二维码两种模式的状态切换复杂** — `scanMode`, `showResult`, `showQrResult`, `result`, `qrResult` 多个状态变量管理，容易出现状态不一致 | **P2** | 交替使用拍照识别和扫码 |
| 7 | **`editField` 的数据绑定使用 ES6 模板字符串语法但实际是字符串拼接** — `this.setData({ [`result.${field}`]: value })` 这种写法在微信小程序中可能不被支持 | **P2** | 识别后编辑任一字段 |

## 六、综合评分

| 维度 | 评分 (1-5) | 说明 |
|------|-----------|------|
| 技术创新 | ★★☆☆☆ | AI对话/生成/扫描均为Mock模拟，无真实AI能力接入，对CEO关注的技术创新不够 |
| 产品差异化 | ★★★☆☆ | 功能矩阵覆盖面广（对话/生成/扫描/匹配/洞察），但深度不足，多为静态演示 |
| AI能力展示 | ★★☆☆☆ | 多数AI功能使用 setTimeout+Mock 模拟，缺少真实大模型推理、OCR识别、智能匹配能力 |
| 用户体验 | ★★☆☆☆ | 存在P1级阻断问题（输入框预填不显示、picker索引未初始化、页面未注册） |
| 功能完整度 | ★★☆☆☆ | AI中心功能入口多但多为空壳，"盖娅进化大脑"等页面代码缺失 |

**⚠️ 关键风险**:
1. AI生成页的输入框预填信息未绑定（**P1阻断**）——用户看不到自己已保存的信息
2. 匹配页的 picker 索引未初始化，筛选功能异常（**P1阻断**）
3. AI中心的gaia/feedback/config页面路径可能不存在（**P1阻断**）
4. 扫描页的 `app.getState()` 调用方法不存在（**P1阻断**）
5. AI对话无法自动滚动到底部（**P1**）
6. 所有AI功能均为Mock模拟，无真实AI能力接入（产品核心差异化缺失）
